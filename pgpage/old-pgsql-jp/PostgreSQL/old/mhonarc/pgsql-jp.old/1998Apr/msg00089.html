>昨年末からMLに登録させていただいておりましたが、最近ようやく使えるマシンがで
>きて
>Postgresqlにトライしようとしているところです。
>
>実はインストールがまだ上手くいってないのですが、そちらは自分で勉強することに
>して、
>Postgresqlの機能や能力という部分で質問させていただきます。
>
>私は、ファイルメーカー Proというソフトでの開発を手掛けていまして、最近このソ
>フトが
>CGI、httpd機能を内蔵したことから、Webデータベースの開発も守備範囲になりました。
>ところが、ファイルメーカー Proにはいくつか難点もあって、例えば、
>
>１）CGI経由のアクセスだとレコードロックが効かないので同時参照・更新が行われ
>ると、
>　　最後の書き込み結果だけが実際に反映される。

これは PostgreSQL なら大丈夫です、

>２）マルチスレッドに対応していないので、複数ユーザからアクセスがあった場合応
>答が
>　　順番待ちになってしまう（前のユーザが時間のかかる処理を行っていると全員が
>待た
>　　されてしまう）。

設計によります。PostgreSQL では、テーブル単位でロックをかけるので、一
つのテーブルに書き込みが集中し、かつ処理に時間がかかるようだと応答が悪
くなります。

>３）内蔵のCGI機能以外の付加機能を付けられない（Perlなどと併用できない）。
>
>などです。

もちろん PostgreSQL なら大丈夫です、

>また、Postgresql6.3.1 + PHP/FI + Apacheでショッピングカートのようなシステム
>を構築
>して、処理速度やセキュリティの面での不安というのはありませんでしょうか？

性能に関しては、使うハード(メモリ、ディスク etc. etc....)がわからない
し、どの程度の要求かも分からないのでなんとも答えようがありません。
# 具体的に「XXX 万件のデータから N 個検索するのに Y 秒以内」とか目標がはっ
# きりしてないと...

また、RDBMS ならどれでも同じことが言えますが、テーブル設計や SQL の組
み方で天と地程の性能の差が出ます。つまり、チューニング次第だということ
です。

セキュリティに関しても同様で、要求によりますね。その気になれば、
Kerberos とかでかなり強固なセキュリティをかけられますし(もっとも、
web で使えるかどうかは知りません)。

いずれにしても、ミッションクリティカルなシステムが必要であれば、プロト
タイプを作成するなどして慎重な検討を行なう必要があるし、もしノウハウを
貯める時間がなければコンサルタントをやとうことをお勧めします。

# 逆に言うと、時間さえあれば、じっくりニツメテ行くことも可能ではないか
# と。
