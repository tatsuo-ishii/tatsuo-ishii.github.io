>なるほど勉強になります．すると次に問題になるのは，

>        2. 各国 (日本であれば JIS) レベルで決めるのか，
>        3. ISO のレベルで一応の合意を作るのか，
>ということですが，結局 ISO Standard が

そうですね。ISO levelで naming convention を決めて、名前の登録そのもの
は、別団体でやるというのが流れみたいですね。丁度ECMAでcharacter setの
レジストレーションをやっているみたいに。でもって、Internetにはいろいろ、
その手のプロシージャがあるので、それを参考にすればいいのではないかなと
思ってます。ただcollationの登録というのはまだ標準化されていないような
ので(本当かな)それもあわせておこなう必要ありそうです。

>> あと余談ですが、SQL-92では実装定義と実装依存を厳密に区別していて、
>>          m) implementation-defined: Possibly differing between SQL-
>>             implementations, but specified by the implementor for each
>>             particular SQL-implementation.
>> 
>>          n) implementation-dependent: Possibly differing between SQL-
>>             implementations, but not specified by this International
>>             Standard and not required to be specified by the implementor
>>             for any particular SQL-implementations.
>> 
>> となってます。character setの定義はimplementation-defined(実装定義)で
>> す。
>
>としているということは，当面は (1) で突っ走って混乱は (2) でなんとか収拾する，
>が ISO のスタンスということなのですね．

わたしの理解では2)というのは、例えば、query optimaizerの実装をどーする
かというようなもので、実装Ａのリリース１とリリース２で実装がことなって
もいいような項目だと思います。1)は一方で、どのように実装したかを明らか
にしなければいけない項目なので、文字型の最大長は１２８バイトであるとか
いうようなものです。おおざっぱに言えば、1)はマニュアルにそれを記さない
といけなくて2)はよい。

いづれにせよ、1)でも2)でも、その機能は実装毎に異って実装されうるわけで
すから、ポータビリティをそこないうれしくないです。character setの場合、
実行結果が異る可能性があるわけですね。
