At 6:29 PM 98.11.12 +0900, SHIOZAKI Takehiko wrote:
>塩崎です。

>>これらのどれかを読むとお分かりかと思いますが、非常に計算コストのかかる
>>最適化手法なので、事前にパラメータを最適化するのに使われて、ユーザが何
>>かするときにはすでに最適化されたパラメータを用いるような使われ方が一般
>>的です。データベースのように、リアルタイムに応答するアプリケーション内
>>で使われるのは、珍しいと思います。

いくら時間がかかっても、良い結果が出るのなら、計算ずみの query plan を
セーブしておいて再利用するようなことも考えられるのですが、途中でメモリを
食いつぶすのではそういうわけにもいかないですしね。

しかし、だからといって、geqo を使わなければうまくいくかというと
そんなことはなくて、やっぱりメモリを使い果たしてしまうので、
結合の度合いが多い場合には geqo の世代数を低くするなりして
現状では geqo を使っていくしかないような気がします。

# 世代数が小さい geqo って、乱数で適当にプランを決めるのとたいして
# 違わなかったりして:-)

>たしかに、6.3.2の頃はパラメータをうまく設定できなかったからかもしれませ
>んが、やたらと遅くてかえって使いものにならなかった記憶があります。
>多くの表を一気に結合するよりは、(一時表を作りながら)少しずつ結合した方が
>ずっと速かった気がします。

ところで、多くの表を結合するための最適な query plan を得ようとすると、
組み合わせの数が爆発して実質的に計算不能になると聞いた憶えが
あります。geqo はそれをなんとかしようとしているのだと私は
解釈しているのですが、商用RDBMSなどはどうやって対応している
のでしょう？もしかしてヒューリスティックの固まり？
