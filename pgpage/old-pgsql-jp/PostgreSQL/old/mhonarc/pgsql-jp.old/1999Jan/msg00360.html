>私の方では、仕事では、外部オブジェクトへの参照を管理、個人的
>には、Laege Objectを使用しています。Webベースの場合だけを考え
>れば良いのなら、外部オブジェクト参照が利点が多いと思いますが、
>ひとつのデータベースをさまざまな方法で利用する場合には、バイ
>ナリデータもデータベースの中に持った方が、itoさんも発言して
>いる通り整合性が取り易くなります。この場合の問題点はデータが
>多くなった場合のパフォーマンスの低下(ファイルの場合でも同じ
>ディレクトリに入れれば一緒?)と、pg_dumpが対応していないこと
>です。
>
>例えば、pg_dumpをLarge Object対応にするというのは、大それた
>ことなのでしょうか。問題になるとすれば、それぞれのファイルを
>どこに置くかという点位ですか?

1. large object を列挙する方法がない:-)

xinv* で始まるテーブルが large object だと決めつけ、その下の数字をoid
であると見倣すことはできます。その場合の問題点は...良く分かりません。

2. pg_dump の出力から large object を復元する際に、oid を指定すること
ができない。おそらく large object を使うアプリケーションでは、large
object の oid を他のテーブルに持っていると思うのですが、復元後その oid 
と整合性が取れなくなります。
