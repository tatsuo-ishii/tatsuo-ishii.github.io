> Web ページに書いてあるのは 受け取った SQL 内の対象部分 (where 条件の値)に
> 対して、暗号化処理を行なって、その値で書き直した SQL を PostgreSQL に送る
> ような中間層があれば、可能です。と言ってるのです。当然後藤さんもこんなこ
> とは承知の上で where 句の対象になり得ない とおっしゃってる(と思う:笑)ので
> すが、なんせ、検索を可能にするためには統一キーワードで暗号化しなくてはな
> りませんから、

基本的にはそういう事です。

一応、この製品でいう「文字列単位」方式であれば完全一致検索なら OK とか、
「シリアル」方式なら元のデータに対する順序性に変化はないとありますが、
これらの場合であっても、検索条件に制約がかかる事には変わりありません。

ただ、これらの方式の場合は暗号強度が落ちるそうなので、暗号強度を
気にしているらしい元記事の方がこれを採用するかどうか… という点も
どうなんでしょうかね。

また、このように暗号化されたカラムデータを格納した場合は、データベース
管理者自身も psql などの標準ツールを使っての当該カラムに対する条件検索が
できないので、開発中やメンテナンス時などで何かと不便な事があります。

運用上のこのような不便が問題にならないとか、その不便をしのぶだけの
価値があると判断する人が採用するのでしょう。

> > そんな事を気にするくらいなのであれば、crypt (3) を使ってデータを暗号化
> > するアプリケーションを書いた方がいいんじゃないですか？
> という結論になるわけです。
> 
> 最終的には、Webサーバー - DBサーバー間の暗号化ならいざ知らず、「DB 内に暗
> 号化されたデータを Insert しても良いって言うのなら、不安に思う暗号化アル
> ゴリズムは使わなきゃいーじゃん？」ってことですね。

そうです。
そういえば、元記事の方はそもそも Web サーバ <-> DB サーバ の間の通信文
そのものの暗号化を行なう方法を探していたのではないのでしょうか？

> 案外、ビットシフトさせるだけのプログラムとかの方が、今日びのクラッカーに
> は効果あったりして・・（ない？汗）
