> に対して暗号化を施して、暗号化データそのものをDBに格納するためのツールのよう
> です。

これを PostgreSQL アプリケーションで使うという事は、つまり当該テーブルの
当該カラムにデータそのものを暗号化した形で格納するという事ですね？

この場合はそのデータそのものに対する検索… つまり、SQL の where 句の
対象とはなり得なくなりますが、それで運用上問題とはならない使い方を

そんな事を気にするくらいなのであれば、crypt (3) を使ってデータを暗号化
するアプリケーションを書いた方がいいんじゃないですか？

それなりの実績はあるという事にもなるでしょう。

アプリケーションを Java で開発する予定なので、C から使うライブラリは…
とかいう話なのであれば JNI 経由で使えばいいだけなので、たいした問題
にはならないと思いますけど。
