今手元にある資料では (すこしふるいのですが)、
「マルチバイト泣き分かれ問題」は発生しないことになっているようです。
「INSERT 文字あふれ」については、「実装依存」なのだとおもいます。


    文字集合名の定義

	CARACTER (<長さ>) CHARACTER SET <文字集合指定>

	  文字集合は、 ASCII、EBCDIC、JIS 0208 など。
	  長さは文字の個数。

	NATIONAL CARACTER (<長さ>)

	  処理系定義の既定の各国文字列型

    CARACTER の演算

	SUBSTRING("abcdef", 3, 2)	   →	"cd"
	SUBSTRING("漢字の文字列", 4, 2)	   →	"文字"

        LENGTH CHAR_LENGTH("漢字の文字")   →	5
	LENGTH OCTET_LENGTH("漢字の文字")  →	10
	LENGTH BIT_LENGTH("漢字の文字")    →	80


# そういえば、INFORMIX (バージョン 7) の NCHAR 型は、
# マルチバイトではなく、シングルバイトだった様な気がする (記憶曖昧)。
#
# CHAR 型がシングルバイトの場合、SUBSTRING() も
# 「マルチバイト泣き分かれ問題」が発生しますね。
# INFORMIX のサーバは、マルチバイトの途中は空白で埋めてくれますです。
