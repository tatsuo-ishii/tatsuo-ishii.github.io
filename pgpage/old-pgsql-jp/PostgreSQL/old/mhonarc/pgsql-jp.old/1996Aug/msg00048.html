IW, May 13, 1996 (http://techweb.cmp.com/iw)

DATABASE SOFTWARE

"Open The Gates To OBJECTS" (By John Foley)

[WHILE UNIVERSAL DATABASES MAY UNITE MULTIMEDIA AND LEGACY DATA,
VENDORS CAN'T AGREE HOW.]

Universal Database は、マルチメディアおよびレガシデータを結合するが、
やり方についてはベンダー毎にまちまちである。


データベースソフトウェアサプライアたちは、ここ十年間で最大の技術的な難
題に直面している。レガシデータとマルチメディアデータを結合するようなア
プリケーションに駆り立てられて、Informix, Oracle, Sybase, およびその他
は、"universal database" を構築すべく探求している。universal database 
とは、音声、イメージ、テキスト、およびその他の complex なデータ型を、
ユーザが既にリレーショナルデータベースに格納している、単純(plain)な修
飾のない(unadorned)文字列や、数値などと統合するものである。


しかしながら、universal database を作るのは、はなはだしく困難であるこ
とが判明した。開発プロジェクトの中のいくつかは、スケジュールから数ヶ月
遅れとなり、他のいくつかは、完全にスクラップにされたものもある。頓挫し
たベンダーたちは、問題を解決するための最善のやり方について口論を始めて
しまった。その一方で、ユーザーたちはその真中に捕われている。「我々の 
IS 担当者は、さまざまな選択肢を見つめながら、どれが最善なのかを判定し
ようとしている」と Bill Rollinson (WWW 上の 小売業者である、Internet
Shopping Network の共同設立社およびマーケッティングの VP) は言う。「私
は今の所では(どれが最善か)分らないと思う」


分っているのは、universal database に対する需要が非常に大きい(ものにな
る)かもしれないということである。Illustra Information Technologies (オ
ブジェクトリレーショナルデータベースを最初に構築した会社の一つ)の公式
の報道担当は、universal database の市場は今日のリレーショナルデータベー
スの市場より大規模なものに成長すると信じていると語る。「人々は、文書、
HTML ページ、イメージ、音声ファイル、ビデオクリップ、およびマルチメディ
アコンポーネントをデータベースの中で管理することができるようになり、
Oracle, Sybase, または Informix で慣れ親しんでいるような、スケーラブル
で、柔軟なアクセスが可能となることを望んでいるのだ」と Malcolm
Colton (Oakland, Calif. にある Illustra のデータベースマーケティングの 
director) は言う。Mike Kennedy (Burlingame, Calif. にある Meta Grroup 
のアナリスト) は加えて言う：「何か現実的なアプリケーションを提供(用意)
することに対して圧力が高まっている」


universal database はどのように使うことができるのだろうか。Internet
Shopping Network (Palo Alto, Calif.) は、それを使って、Hammacher
Schlemmer および FTD などの顧客に対して、現在小売を行っている 35,000 
の製品を販売するのを助けたいと考えている。今日、この会社のサイバースト
ア(cyberstore)では、リレーショナルデータと、付属して現われる製品の小さ
な写真のみに限られている。しかし、将来の取引では、もっと豊か(rich)なデー
タを組合せたもの−−および、それらを取り扱うことができるようなデータベー
ス−−を絡ませることになるだろう。「音声をもっと増やし、もっと多くのアニ
メーションを入れて、もっと個人化することになるだろう」と、共同設立者の 
Rollinson は言う。「そういう所こそ、バックエンドのデータベースが役に立
つのさ」


Rollinson およびその他の、まさにマルチメディアデータベースを実現しよう
としている(実現する準備をしている)人々にとっての問題は、ベンダーたちが
まだ準備ができてないということである。「実現方式の選択肢はたくさんある」
と、Jnan Dash (Oracle のデータベース技術の VP) は言う。「気が遠くなる
ような難題さ」


遅れの一つの原因は、universal database を構築するための最善のやり方に
ついて、ベンダー間で合意がないことである。ほとんどのベンダーは、二つの
アプローチのうちの(どちらか)一方に(分れて)列をなすことになった。一つの
グループは、既存のリレーショナルデータベースを、オブジェクト指向の拡張
を用いて改装することを好み、いわゆるオブジェクトリレーショナルデータベー
スを作成した。もう一方は、オブジェクトのみを取り扱う新たなデータベース
を構築するのを好む。論争は解決からは程遠い。つい先週のこと、Sanjay
Kumar (Computer Associates の president および chief operating
officer) は、Boston での Object World コンファレンスの基調演説を使って、
オブジェクトリレーショナルデータベースは、ビジネスユーザーの問題を解決
することはできないと主張した。


IBM, Informix, および Oracle は、オブジェクトリレーショナルのルート(道)
を進んでいる。CA も、最初はその道を進んだが、結局その努力を捨てる結果
になった。CA は、現在では、顧客に対して、リレーショナルおよび非リレー
ショナルデータを別々のシステムに格納するように薦めている。Microsoft は、
OLE DB (データを、Microsoft の SQL Server データベース、Windows NT オ
ペレーティングシステム、および Winodws アプリケーションに対してばらま
く(散在させる)ことを可能にするインタフェース)を頼りにしている。Sybase 
は、あらゆる種類の complex データを単一のシステム内で統合する努力はし
ないであろう。その代わりに、いくつかのタイプのデータに対しては、専用
(special-purpose)のサーバ群を、ミドルウェアで接続して、提供することに
なるだろう。


自分の(使っている)リレーショナルデータベースベンダーが選択したアプロー
チとは異なるオブジェクトのアプローチを好ましいと考えるユーザにとっては、
選択することは困難になるであろう。例えば、Internet Shopping Network の 
Rollinson は、もし complex データが専用サーバ群に格納されたら−−(つま
り) Sybase によって提唱されている戦略を採れば−−ベストの性能になるか
もしれないと信じている。しかし、彼のデータベースサプライヤである 
Informix は、オブジェクトリレーショナルモデルを追及しているのだ。


議論の背景にあるのは、企業データベースの中に、非リレーショナルのデータ
を取り込みたいという願望である。しかしながら、リレーショナルデータベー
スは、正直なところ制限されたやり方ではあるものの、complex データをすで
に扱っている。例えば、写真や短い音声などは、多くのデータベースでは、
Blob (binary large object) として取り扱うことができる。問題は、Blob が
「鈍感」(opaque)であることであり、Blob の中に含まれるデータについては
管理(操作)したり検索したりできないということを意味する。「Blob の場合
には、データベースが Blob の中身を見ることができないので、何かを『探し
て来い』と言うことができないのです」と、Illustra の Colton は言う。


オブジェクトリレーショナルデータベースは、そのような問題を克服すること
を約束している。現在は、Illustra, OmniScience, および UniSQL などの少
数の小さな企業が販売しているが、これらのハイブリッド製品は、実質的に企
業システム内のあらゆる種類のデータを取り扱うことが可能になる。「どんな
ものでも可能です」と、Dick Williams (Illustra の president, Illustra 
は、12月に Informix により買収された)は言う。「もし想像することができ
て、定義することができれば、それをカプセル化してデータベース上に置くこ
とができるのです」


オブジェクトリレーショナルのアプローチが有意義だと感じるユーザも存在す
る。「これで私の仕事が簡単になる」と、Henry Kucera (British Columbia 
州の環境・土地・公園省のデータ管理のトップ)は、言う。「容易に管理する
ことができない、たくさんの独立したコードを開発していくというのは好まし
くないことです。そういうことをすると論理の悪夢となります」


オブジェクトオンリー(オブジェクトのみ)のルート(道)を進んでいる企業とし
ては、ObjectDesign (Burlington, Mass.), O2 Technologies (Moutain View) 
および Versant (Menlo Park, Calif.) などが含まれる。しかしながら、専門
家や競合他社の中には、これらの製品群は、企業規模での展開に必要とされる
ハイエンドの能力(機能)を欠いている、と言う者もある。


Filling The Void

大手のデータベースサプライヤは、これらの問題を解決したいと考えている。
２月には、Oracle は、Oracle Universal Server をアナウンスした。これは、
Oracle のリレーショナルデータベースの新リリース(Oracle 7.3)と、６つの
専用サーバ(special-purpose server)群(Web, spatial, text, video, OLAP
(online analytical processing), および messaging)を統合するものである。
様々な(異なる)データタイプが、別々の物理的または論理的な格納庫(store)
に格納される。「まさに Universal Server が今ここにあります」と、Ken
Jacobs (Oracle のサーバ戦略の VP) は言う。


British Columbia 省は、Oracle 製品の早期ユーザである。この機関は、最終
的には、カナダ政府内の 200 のデータソースに対する公のアクセスを提供す
ることになる予定のウェブサイトを一つ作った。使われている技術の中には、
Oracle のリレーショナルデータベース、Web サーバ, Spatial サーバ、およ
び text サーバが含まれている。


アナリストたちは、(Oracle の) Universal Server は、1990年代の初めから 
Oracle の重役たちが言い続けてきたようなオブジェクトリレーショナルデー
タベースではない、と指摘している。そのような製品である Oracle8 は、い
まだに開発中であり、最も早くても来年までは出てこないであろう、と彼らは
加える。


この理論によると、Oracle は、Informix が年末までに universal database 
を出荷するというプランをアナウンスしたことにより、圧力がかかって、
universal database を導入することにした、ということになる。「私は、
Oracle7.3 は、決して universal database となることを本当に意図したもの
ではなかったと思う」と、Mike Norman (Milton Keynes, England にある 
Bloor Research Group の調査局長)は言う。「『the Oracle Universal
Server』というのは、方向性のを述べたものと見るべきであろう」


どちらにしても、このトピックは Oracle での、要注意の課題となってしまっ
た。ある会社のスポークスマンが、最近、チェアマン兼 CEO である Larry
Ellison に対して、Oracle8 に関する疑問を全て問い合わせた。「Oracle は
当惑している」と、Meta Group の Kennedy は言う。


Oracle の見方は違うようだ。Jacobs は、スケジュールから遅れているという
よりも、Oracle は、実際は、出荷スケジュールを速めて、7.3 に (オブジェ
クトリレーショナルではないものの) Oracle8 の要素のいくつかを含めたのだ、
と主張する。「我々は、それなりの確固たる仕事をしたのだ」、と彼は言う。
「ことによると、我々は、Oracle7.3 のことを『Oracle8』と呼ぶべきだった」


一方、Informix の重役たちは(Menlo Park, Calif.)、Illustra を企業買収し
たことにより、universal database を作ると努力している競合他社に対して、
２年間のリードを得たと言う。Informix は、自社のリレーショナルデータベー
スと、Illustra の(DataBlade と呼ばれる)データ拡張(data-extention)技術
とを統合しようとしている。最初のステップとして、二つの製品がこの四半期に
予定されている。Illustra-to-Informix gateway と、Informnix と Illustra 
データベースの両方で動作する DataBlades developer's kit である。
Informix の Universal Server の最初のリリースは、年末より手前に出荷さ
れるスケジュールになっており、Web, images, colors/shapes, および video 
を含む、いくつかの DataBlade は Illustra データベース上で(は)動作する。


２月に Informix の Universal Server がアナウンスされた時、America
Online, Deutsche Telekom, および Coldwell Banker を含むいくつかの企業
が、それを支持した。John Parady (Mission Viejo, Calif. にあるColdwell
Banker の CIO) は、このシステムは、Web 上で不動産を売るのに使うことが
できると信じている。「究極のポイント(目標？)は、家の中を仮想的に歩き回
ることだ」と彼は言う。


しかしながら、Informix と Illustra の製品群をマージするというのは、甚
大な技術的努力を要するものであり、これについて、アナリストたちは、予期
されているよりも長い時間を要するかもしれないと警告している。この作業に
は、100万行の Informix のコードと、50万行の Illustra のコードを統合す
るということが絡む。「控え目に言っても、Informix のエンジニアたちは、
来年になっても、やるべきことは山ほどある」と、Charles Phillips (New
York の警備会社である Morgan Stanley のアナリスト) は言う。「このソー
セージを作ろうとしているやり方の詳細を見ることによって、我々は、一般に
評価されているよりも、仕事は困難で複雑であるということを確信するであろ
う」結論として、Phillips は、production-ready (?) な製品は、来年の第２
四半期までは見込めないとしている。Oracle の Ellison はもっと厳しい。
「彼らは、決して、絶対に、Illustra と Informix を統合しないだろう」と
彼は言い捨てる。「そんなのは馬鹿げている。これは今までに私が聞いた中で
最も愚かなことだ」


Sybase (Emeryville, Calif.) は、自分の戦略を発展させ続ける。この会社は、
昨年、オブジェクト指向のデータベースを開発するプロジェクトをスクラップ
にした。５月６日、Sybase は、Adaptive Server (オブジェクトリレーショナ
ルデータベース)をアナウンスした。Sybase は、あらゆる種類の complex デー
タをリレーショナルデータと統合するということはしない。いくつかのデータ
タイプについては、今迄通り専用のサーバ群によって取り扱われるであろう。
「我々は、スイス軍用ナイフを売ろうとしているわけではない」と、Dan Lahl
(Sybase の サーバマーケティングの director)は言う。「異なるアプリケー
ションには、異なるデータベースが必要となる」Dennis McEvoy (Sybase の企
業グループの president) は加えて言う、「完全な統合というのは我々のトッ
プの優先事項の一つではない。Universal Server というのは、Universal
Database Disaster になる可能性を持っている」


他のデータベースサプライヤたち(特に注目されるのは、IBM, Microsoft, お
よび Computer Associates)も乱闘に加わってきた。IBM は、自社の DB2 デー
タベースに対して拡張部(extensions)を作成している。テキストに対するサポー
トは１月から available となっており、image, audio, および video 用の 
extension はβテスト中である。さらに多くのものが予定されている。time
series エクステンダ(株式の取引価格などの、シーケンシャルな、時間(時系
列)に依存するデータを格納するのに使われる)は、現在開発中である。DB2 の
エクステンダは、まずは、IBM の OS/2 および AIX システム上で available 
となる予定である。Windows NT, 他の UNIX, および IBM のメインフレームの 
OS 上の DB2 に対してもサポートが計画されている。


complex データの問題に対する Microsoft の答えは、OLE DB である。これは、
様々なデータタイプ群 (E-mail, ドキュメント、写真、およびその他の、SQL
Server データベース, Windows NT ファイルシステム, または Microsoft 環
境の他の場所に格納することができるコンポーネントなど)を扱うための、オ
ブジェクト指向のプログラミングインタフェースである。「OLE DB は、デー
タを、どこにあるかに関係なく、得るための機構である」と、Dan Basica
(SQL Server の product manager) は言う。


Computer Associates は、さらに別の針路を取りつつある。CA は、初めはオ
ブジェクトリレーショナルデータベースを構築しようとした。その後、昨年、
それは努力に値しないと決断した。「両方の世界の中で最悪のものを作り出す
ことになると悟ったのです」と、Yogesh Gupta (製品戦略の senior VP)は言
う。「基本的な問題は、リレーショナルアーキテクチャは(すでに)その限界ま
で拡げられており、新しいアプリケーションに対しては適当ではないのです」


CA の選択肢は、Jasmin である。Jasmin は、Fujitsu によって開発されたオ
ブジェクト指向のデータベースであり、ゲートウェイと SQL ライブラリを通
じて、CA の OpenIngres の内部に接続されている。「新たなアプリケーショ
ンはより効率的になるでしょう、そして既存のアプリケーションもより効率的
になるでしょう」と Gupta は言う。キャッチ[フレーズ]：オブジェクトデー
タベースとリレーショナルデータベースの両方からデータを引出すようなアプ
リケーション(で)は、性能が悪くなることがあります。


業界のアナリストたちは、どのアプローチ(オブジェクトリレーショナル、オ
ブジェクト指向、ミドルウェア、OLE DB、またはその他)がトップに登るであ
ろうか、を予測するには時期尚早であると言う。たぶん、どれがトップになろ
うとも、IS の希望の的となることだろう。

HE DOES OBJECTS
(Michael Stonebraker (オブジェクトリレーショナル技術の父)は、業界が変
化の開始点にあると見ている)


Michael Stonebraker (Informix Software Inc. の chief technology
officer) は、データベース業界は、オブジェクトの方へ動きつつある、と言
う。証拠として、彼は、世界で最も大規模なデータベースプロジェクトが、彼
の作ったハイブリッドソフトウェア上で着手されている、ということを指摘す
る。


Stonebraker は、オブジェクトリレーショナル技術の父であると広く見なされ
ている。1980年代の終りに University of California, Berkeley 校 の教授
であった時に、彼は、学生達と一人のフルタイムのプログラマから成るチーム
を率いて、Postgres と、オブジェクトリレーショナルデータベースを開発し
た。「universal database に関して、彼と彼の学生達はハードな指向をたく
さん重ねた」と、Curt Monash (New York の software consultant) は言う。


Stonebraker は、1992年に、パブリックドメインソフトウェアである 
Postgres を商用化するために、Illustra Information Technologies を設立
した。Stonebraker は、Postgres について、C++ または Smalltalk などの第
３世代言語よりはむしろ、SQL プログラミング言語に基づいたオブジェクト指
向データベースであると説明している。


研究用のソフトウェアから商用の製品を作り出したのは、これが最初ではない。
以前に、リレーショナルソフトウェアを販売するために、Ingres を共同設立
した。(Ingres は、Ask Group が買収し、その後、Computer Associates に買
収された) 「基本的な問題は、学術的な研究所では、バグのあるプロトタイプ
が構築され、ドキュメントはお粗末で、あまり高速ではない、ということだ」
と、Stonebraker は言う。「それを格好のついたものにするためには、商用ベ
ンダの背後にあるお金が必要になるんだ」


Illustra は、1994年に売り出され、月に15％の割合で成長している。NASA の
ための仕事をしている Hughes Information Technology Systems が、衛生に
基づいたグローバルなイメージング(映像処理)システムを展開するプロジェク
トにおける、巨大な基盤(underlying)データベースとするために、Illustra 
の製品を取り上げた時に、大きな分岐点を向かえた。そのデータベースは、１
petabyte (または、1,000 terabytes) にまで膨らむことが見込まれている。
他の機関でも Illustra の胴まわり(格納容量の意か？)を増やすように圧力が
かかっている。「彼らは、『分散機能はどこにあるんだい』『5,000ユーザま
でのスケーラビリティはどうですか』『MPP はありますか』と尋ねたんだ」と、
Stonebraker は言う。「我々は、機能(性)はちゃんと備えていたのだけど、ス
ケーラビリティはなしだった」


この前の11月、Informix (Menlo Park, Calif.) の代表は、Illustra に、お
互いの製品をマージするというアイディアを持ちかけた。Illustra は同意し、
12月にその取引についてアナウンスされた。


Informix と Illustra が第４四半期のデッドラインに間に合うということに
ついて懐疑的な見方が拡がっているものの、Stonebraker はプロジェクトは順
調だと言う：「我々は、今年中に製品を持って市場に出ます」
