石井です。

中村さん：
>でも、Index ServerはAND/OR/NEARなどの演算子をサポートして
>いますから、文章処理には（私の所では）欠かせなくなっていま
>す。正式にはNT Serverでしか動かない、というのが難点ですが。
>（ちなみにNT Workstationでも動かす事ができてしまいます。）

済みません。NEAR って何ですか？

ところで、このような検索は、専用の全文検索エンジン(WAIS とか なまずと
か...)  を使えば可能ですが、中村さんのおっしゃってるのは、もちろんそん
な話ではなくて、あくまで RDBMS から検索エンジンを使いたい、ということ
なんですよね。

一つの方法は、テキストから単語を抽出し、それをテーブルに入れて検索する
手です。これなら普通のRDBMSでもできます。そこそこのスピードで動きます
が、テーブルが巨大になります。たとえば、このMLのメール(5000件程)から単
語を抽出して作ったテーブルは、レコード数が50万を超えてます。

# ちなみに、英語のみですが、テキスト入力時に自動的に単語のインデックス
# を作ってくれるパッケージが 6.4 の contrib/fulltextindex/ に登録されてます。

>そこまでの性能はなくてもいいから、汎用のテキストファイル
>検索エンジンがあればいいかな、と思っております。
>
>受信したメールのinboxとか、newsのスプールとかを、そこそこ
>の性能のIndexがはれたら、なかなかGoodではないかな、と。
>
>汎用テキスト対応で作っておけば、SQL Server -> Index Server
>の関係のように、PostgreSQLとも仲良くさせる事ができます。

前から思っているのは、既存の全文検索エンジンを PostgreSQL に取り込み、
それを PostgreSQL の関数や executor から呼び出せないかな、ということで
す。どなたか真剣に考えている方、いらっしゃいますか？
