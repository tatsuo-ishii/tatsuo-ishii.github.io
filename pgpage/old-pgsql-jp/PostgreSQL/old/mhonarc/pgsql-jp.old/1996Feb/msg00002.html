1.1)	Postgres95 とは何ですか？
x
	Postgres95 は POSTGRES データベース管理システムの改良版で、次世代
	DBMS 研究用のプロトタイプです。Postgres95 は POSTGRES の強力なデー
	タ・モデルと豊富なデータ・タイプ(型)を内包しつつ、POSTGRES で使わ
	れた PostQuelクエリ(問い合わせ)言語を、SQL のサブセットを拡張した
	ものに置き換えています。Postgres95 は無料で完全なソース・プログラ
	ムの利用ができます。

	主な Postgres95 の著者は Andrew Yu と Jolly Chen です。その他、
	たくさんの人々がこのコードの移植、テスト、デバグ、および、改良に貢
	献しました。Postgres95 の派生元のコードである POSTGRES はカリフォ
	ルニア大学バークレー校において、 Michael Stonebraker 教授の指揮の
	もと、たくさんの学生、卒業生、プログラマーたちの努力により作られま
	した。
	

1.2) 	Postgres95 の動作環境は？

	著者らは Postgres95 のコンパイルとテストを次のプラットホーム上で
	行ないました。
	
	- OSF 2.1, 3.0, 3.2/Alpha with cc and gcc 2.7.0
	- Ultrix 4.4/DECStation with cc and gcc 2.7.0 (libdl package required)
	- SunOS 4.1.3/SPARC with gcc 2.7.0
	- Solaris 2.4/SPARC with gcc 2.7.0
	- HP-UX 9.0/HP 9000/700 with cc. (It is possible to compile Postgres95
	  with gcc but the dynamic loader will not work.)
	- Linux/Intel x86, ELF format:  kernel 1.2.10, gcc 2.6.3 

	また、下記の移植版が Postgres95 の配布に含まれています。著者らはこ
	れらのプラットホームを手近で使えませんが、他の人々の手によりこれら
	の移植版のテストが行なわれました。

	- BSD44_derived/i386. Tested on NetBSD 1.0.  Should also work
	  for FreeBSD and BSD/OS.
	- AIX/IBM RS6000 
	- IRIX 5/SGI. Tested on IRIX 5.3.

	[訳注: Windows-NT版への移植は Jon Forrest(forrest@cs.berkeley.edu)
	さんにより行なわれているそうです。
	  Linux/Intel x86, ELF format は JE4(kernel 1.2.13, gcc2.7.0)でも
	動きました。β0.02 は Linux/Intel x86, a.out の kernel 1.2.8,
	gcc2.6.3 で gnu dld-3.2.7 を使って動きました。
	  日本語への移植は SRA Inc. の石井達夫(t-ishii@sra.co.jp)さんによ
	り行なわれました。このパッチは：
		ftp://ftp.sra.co.jp/pub/cmd/postgres/postgres95-1.0/
	から入手できます。]

1.5)	Postgres95 のためのサポート

	Postgres95 のための公式なサポートは、*ありません*。そのサポートは
	ボランティアの努力を通じてのみ維持されています。

	Vnet の Jason Wright さんらのご厚意により postgres95 メーリング・
	リストは今、新たなホーム -- postgres95@postgres95.vnet.net を得る
	ことができました。Postgres95 に関することであれば、バグのレポート
	や修正版だけに限らず議論ができます。このリストの定期購読の仕方につ
	いて知りたければ、電子メールの(Subject 行ではなく)本文に：
		help
		info postgres95
	と書いて、
		majordomo@postgres95.vnet.net
	に送って下さい。

	ダイジェスト版のメーリング・リストもあります。このリストの購読は
	"本文"に：
		subscribe postgres95-digest
	と書いて majordomo@postgres95.vnet.net に電子メールを送って下さい。
	ダイジェスト版はメインのリストが 30k 前後のメッセージを受信した毎
	にそのリストのメンバーに送付されます。

	[訳注: 日本語のメーリング・リストは石井@SRAさんが管理しています。
	このリストの購読は、本文の１行目に：
		subscribe postgres95
	と書いて majordomo@sramhe.sra.co.jp 宛に電子メールを送って下さい。]


	アーカイブおよび Vnet での新メーリング・リストのダイジェスト版は次
	の場所でみつかります。：
		ftp://postgres95.vnet.net/pub/postgres95/archives
		ftp://postgres95.vnet.net/pub/postgres95/digests
	
	Postgres95 についてもっと詳しく知りたければ、下記の postgres95 WWW
	ホームページからたどれます。：
		http://s2k-ftp.cs.berkeley.edu:8000/postgres95
	

	また、そのミラーサイトは下記の場所にもあります。：
 		http://www.vnet.net/postgres95/

1.6)	Postgres95 の将来のリリースは？

	最も新しい postgres95 のリリースは version 1.0 です。将来のリリー
	スの日取りは未定です。

	[訳注：現在 version 1.01 のα版がテスト中です。]


1.7)	Postgres95 の商用版はありますか？

	Illustra Information Technology が POSTGRES を基にした Illustra と
	いうオブジェクト-リレーショナル DBMS を販売しています。Illustra の
	見かけは postgres95 と似ていますが、もっと多くの機能を持っていて、
	より堅牢で、より性能も良く、実際にドキュメントとサポートを提供して
	います。その反面、費用がかかります。これ以上の情報を得たければ、
	sales@illustra.com に問い合わせて下さい。
	

1.8)	Postgres95 にはどのようなドキュメントがありますか？

	ユーザ・マニュアルと幾つかの小さなテスト例が配布に含まれます。現在、
	それ以外には添付ドキュメントはありません。(もしあなたがドキュメン
	トを書くのを手伝ってもかまわないというのなら、どうぞ著者に連絡して
	下さい。)

	[訳注：POSTGRES4.2 の参照マニュアル(ref.*)が参考になるそうです。
	Postgres95 のリファレンス・マニュアルはもうすぐリリースされるみた
	いです。]
	

1.9)	Postgres95 はどのバージョンのSQLを使いますか？

	Postgres95 は SQL-92 のサブセットをサポートしています。それは最も
	大切な構造は持っていますが、機能の幾つかが欠けています(例えば、
	GROUP BY で使われる HAVING 句)。もう一つの大きな違いは、postgres95
	にはネスト(入れ子)したサブクエリ（副問い合わせ）が欠けていることで
	す。
	
	その反面、ユーザー定義のタイプ(型)、ファンクション(関数)、インヘリ
	タンス(継承)などを創ることができます。

	
1.10)	Postgres95 は以前のバージョンの postgres のデータベースを使って動きますか？
	
	いいえ、Postgres95 v1.0 はそれより前の POSTGRESのバージョン、例え
	ば v4r2 とは互換がありません。実際は、Postgres95 のβリリース
	の 0.01 と 0.02 とさえ互換がありません。そこで、我々は古いデータベー
	スの内容をダンプするのを補助するために 'pg_dump' というユーティリ
	ティを用意しましたが、それでも限界があります。(たとえば、POSTQUEL
	ファンクションを SQL に翻訳したりはしません。)


	
1.11)   何人位の人々が Postgres95 を使ってますか？

	我々はライセンス発行も登録制度もしていないので、それを述べることは
	不可能です。が、数百の postgres95 v1.0 のコピーがダウンロードされ
	たことと、約400名のメーリング・リストの購読者がいることはわかって
	います。

第２節：	インストールに関する質問
----------------------------------------------------------------------

2.1)	Backend をコンパイルしようとすると "cpp: command not found" と
	いうエラーを受け取りました。

	あなたのサイトでの cpp へのパスを含むように
	src/backend/utils/Gen_fmgrtab.sh スクリプトを修正して下さい。
	

2.2)	Backend をコンパイルしようとすると 'yy_flush_buffer undefined'
	を受け取ります。

	もっと新しいバージョンの flex (2.5.2 以降)を使って下さい。

2.3)	initdb が走りません。

	* 適切なパスが通っているかを確認して下さい
	* ユーザ 'postgres' がファイルのすべての権利を所有しているかを確認
	  して下さい。
	* $PGDATA/files の下にファイルがあって、それらが空でないかを確認し
	  て下さい。もし、ファイルが空でなければ、"gmake install" が何らか
	  の理由で失敗しています。


2.4)	Postmaster を起動したら、次のメッセージを受け取りました。
		"FindBackend: could not find a backend to execute..."
		"postmaster: could not find backend to execute..."

	おそらく、パスが正しく設定されていません。'postgres' 実行モジュー
	ルが設定されたパスの中にある必要があります。

2.5)	システムがコンマと小数点と日付フォーマットとについて混乱しているようです。
	
	あなたのロカールの設定を確かめて下さい。Postgres95 は postmaster
	プロセスを走らせたユーザーのロカールの設定を使います。これらをあな
	たの作業環境に合わせて設定して下さい。


2.6)	どのようにすれば /usr/local/postgres95 以外の場所にインストールできますか？

	src/Makefile.global の中のパスを手で修正して、あなたのサイトの
	配置に合わせる必要があります。


2.7)    Backend のコンパイルは成功しましたが、libpq のコンパイルは：
		"libpq/pqcomm.h" not found when compiling fe-auth.c
	と文句を言われて終ってしまいました。

	おそらく、 postgres95 を /usr/local/postgres 以外のどこかにインス
	トールしていますが、それに合わせて src/Makefile.global を直してい
	ません。
	質問 2.6 を参照下さい。


2.8)	Postgres95 のバグの修正版はどこでみつかりますか？

	バグリストとパッチの類は次のウェブ・ページにリストされてます。
	   http://s2k-ftp.cs.berkeley.edu:8000/postgres95/www/pglite5.html

	パッチはリストされている順番に当るべきです。パッチ・ファイルは直接
	次のディレクトリから FTP できます。：
		   ftp://s2k-ftp.cs.berkeley.edu:8000/postgres95/www/bugs


2.9)	すべて問題無く見えるのですが、パッチがうまく当たりません。

	もし、ウェブ・ブラウザから直接カット＆ペーストをすると、タブと空白
	が適切には保存されません。カット＆ペーストの代わりに、ウェブ・ブラ
	ウザの 'save as file' オプションを使って下さい。

第３節：	Postgres95 の特徴についての質問
----------------------------------------------------------------------

3.1)	カラム(欄)に「キー」とかその他の拘束をどのように設定すればよいですか？

	Column constraints are not supported in postgres95.
	As a consequence, the system does not check for duplicates. 

	Postgres95 ではカラムの拘束はサポートされていません。ですので、シ
	ステムは重複のチェックをしません。

	
3.2)	Postgres95 はネスト(入れ子)したサブクエリ(副問い合わせ)をサ
	ポートしますか？

	No, postgres95 does not support nested subqueries.

	いいえ、postgres95 はネストしたサブクエリをサポートしません。


3.3)	ユニーク(重複のない)インデックス(指標)をどのように定義しますか？

	Postgres95 does not support unique indices.  Defining an index does
	not preclude insertion of duplicate index key values.

	Postgres95 はユニーク・インデックスをサポートしません。インデック
	スの定義において、重複したインデックス・キー値の挿入を妨げません。
	

3.4)	ルールを使うといっぱい問題がおきたのですが？
	
	現在、postgres95 のルール・システムはおおよそだめです。このルール・
	システムはビュー・メカニズムをサポートするのには十分働きますが、そ
	の程度のものです。Postgres95 のルールは危険を覚悟の上で使って下さい。
	


3.5)	ラージ・オブジェクトの途中にきちんと書き込むことができないみたいです。

	インバージョン・ラージ・オブジェクト・システムも同様に、おおよそ壊
	されます。大きなサイズのデータを保存して、そして、それらを読み出す
	のには十分働きますが、しかし、実装にいくつかの根本的問題が潜んでい
	ます。Postgres95 のラージ・オブジェクトは危険を覚悟の上で使って下
	さい。

	[訳注：インバージョン・ラージ・オブジェクト(Inversion Large
	Object)は POSTGRES からその仕組みだけを引き継いだラージ・オブジェ
	クトの取扱いで、ラージ・オブジェクトを塊に分解したタップルとして
	保存する方法です。詳しくはマニュアルを参照して下さい。]



3.6)	Postgres95 はグラフィカル・ユーザ・インターフェースを持っていますか？
	レポート・ジェネレータは？埋め込みクエリ言語のインターフェースは？

	３つともありません。少なくとも正式な配布には含まれてません。何人か
	のユーザが 'pgbrowse' と 'onyx' を postgres95 のフロントエンドに使っ
	て成功したと報告しています。

	[訳注：pgbrowse は以前の postgres 用に作られた Tcl7.3/Tk3.6 を
	使ったインターフェースですが、それを postgres95 用に変更した
	pg95browse というのがあります。pg95browse は Tcl7.4/Tk4.0
	+TclX7.4 にパッチを当てないと駄目ですし、それでも不安定です。
	(原因が pg95browse か wish かは不明です。) 
	 その他に α版の pg95admin というのがあります。これは 
	Tcl7.4/Tk4.0 + TclX7.4 + TIX-4.0 を使っていて、ユーザ登録や
	グループ登録ができます。
	 Onix は 4GL の汎用の DBMS インターフェースで POSTGRES4.2 にも
	対応していました。]

	
3.7)	Postgres95 のクライアント・アプリケーションはどのようすれば書けま
	すか？

	Postgres95 は libpq という C言語呼出しのライブラリ・インターフェー
	スと、同様に、 libtcl という Tcl-主導型のライブラリ・インターフェー
	スをサポートしています。

	[訳注：C++呼出しの libpq++ もあります。]

	他の人々が、postgres95 への perl インターフェースと WWW ゲートウェ
	イを寄贈しました。Postgres95 のホームページに詳細がありますので御
	覧下さい。

	[訳注：	WWW インターフェースに WDB(A Web interface to SQL
		DataBase) の Wdb-p95 というのがあります。これは perl インター
		フェースの pg95perl5 を使っています。
			(http://www.eol.ists.ca/~dunlop/wdb-p95/)
		 Python のモジュールの PyGres95 もあります。
			(http://www.via.ecp.fr/via/products/pygres.html)
		 Java のインターフェースもあるそうです。
			(http://www.blackdown.org/java-linux.html)
	]


3.8)	他のホストから私の postgres95 バックエンドへアクセスするのを、どの
	ように防げば良いでしょうか？
	

	現在、Postgres95 はホスト主導型のアクセス制御は持っていません。が、
	ユーザの認証に kerberos をサポートしています。あなたは、単純に
	kerberos フラッグを有効にして造り直すだけです。

	[訳注：kerberos は日本では使えません。Version 1.01 にはホスト主導
	型の認証が加わるようです。]


3.9)	pg_group のセットアップはどのようにしますか？
	
	現在、ユーザ・グループを設定する簡単なインターフェースはありません。
	明示的に pg_group テーブルを insert/update しなければなりません。
	例えば次のようにします。：

	jolly=> insert into pg_group (groname, grosysid, grolist)
	jolly=>     values ('posthackers', '1234', '{5443, 8261}');
	INSERT 548224
	jolly=> grant insert on foo to group posthackers;
	CHANGE
	jolly=> 

	[訳注：pg95admin(3.6の訳注参照)を使うと出来たような気がします。]

	pg_group のフィールドは以下のとおりです。：
		groname:  グループ名。Char*16 タイプで純粋な英数字にすべき。
			  アンダースコアやその他の特殊文字をを含めないこと。
		grosysid: グループID。Int4 タイプで各グループ毎にユニークで
			  あるべき。
		grolist:  そのグループに所属する pg_user id のリスト。
			  このフィールドのタイプ(型)は int4[]。


3.10)	バイナリ・カーソルとノーマル・カーソルの厳密な違いは何ですか？

	ノーマル・カーソルは ASCII フォーマットでデータを返します。データ
	はもともとバイナリ・フォーマットで保存されていますのでシステムは
	ASCII フォーマットで取り出すために変換をしなければなりません。それ
	に加えて、ASCII フォーマットはバイナリ・フォーマットよりもサイズが
	大きくなることがよくあります。また、一旦 ASCII フォーマットで返さ
	れたアトリビュート(属性)をクライアント・アプリケーションで扱うため
	に、再びバイナリ・フォーマットに変換しなければならないことがよくあ
	ります。

	バイナリ・カーソルはデータをもとのバイナリ表現で返します。この
	ため、バイナリ・カーソルは変換のオーバーヘッドが少ないので多少
	速くなる傾向があります。

	しかしながら、バイナリ表現は異なるマシン・アーキテクチャ間で異なる
	可能性があるのに対して、ASCII はアーキテクチャに関しては中立です。
	このように、クライアント機とサーバ機が異なるバイナリ表現を持つとき、
	アトリビュート(属性)をバイナリ・フォーマットで取り出すことはあまり
	やりたくないでしょう。また、もし本来の目的がデータを ASCII で表示
	することであれば、ASCII でデータを取り出すとによりクライアント側で
	の作業は減ります。


3.11)	なぜ != 演算子が動かないのですか？
	

	SQL は <> を不等号の演算子として指定していますので、我々はそれをビ
	ルトイン・タイプ(型)として既に定義しています。しかし、!= 演算子を
	組み込んで postgres95 を拡張することはあなたの自由です。


3.12)	R-tree インデックスは何ですか、また、何に使われますか？

	R-tree インデックスは空間的なデータにインデックスを付けるために使
	われます。ハッシュ・インデックスでは範囲の検索ができません。また、
	B-tree インデックスでは、１次元でしか範囲の検索ができません。
	R-tree インデックスであれば多次元のデータを扱えます。たとえば、も
	し R-tree インデックスを 'point' 型のアトリビュート(属性)に付ける
	ことができるとするとシステムは、長方形に囲まれたポイント(点)をすべ
	てを選択するというようなクエリに、もっと効率良く答えられます。

	R-Tree の設計の原典となる権威ある論文は:

	Guttman, A. "R-Trees: A Dynamic Index Structure for Spatial Searching."
	Proc of the 1984 ACM SIGMOD Int'l Conf on Mgmt of Data, 45-57.


	で、この論文は、Stonebraker 教授の "Readings in Database Systems"
	でも取り上げられています。


3.13)	タップル(1レコード)の最大サイズは？

	タップルの大きさは 8K バイトに制限されています。システムのアトリビュー
	ト(属性)やその他のオーバーヘッドを考慮すると、安全側に見積もって１
	タップルあたり 8,000 バイトにとどめるべきでしょう。8K を越えるアト
	リビュートを使うときは、ラージ・オブジェクト・インターフェースを使っ
	てみて下さい。
	[訳注：8K は 8*1024=8192]


3.14)	カラム(欄)を NULL にアップデートするにはどうしますか？

	現在、それを行なう術はありません。


3.15) 複数のインデックスを定義しましたが、私のやったクエリではそれらを使っ
ている様子がありません。なぜですか？

	Postgres95 は統計情報を自動的には維持しません。統計情報を更新する
	ためには、明示的に 'vacuum' を呼び出さなくてはなりません。統計情報
	が更新された後は、オブティマイザ(最適化モジュール)がインデックスを
	使うようになり的中率が上がります。ただし、オブティマイザには制約
	があって、(OR 句のような)いくつかの状況ではインデックスを使いませ
	ん。


3.16)	Postgres95 のための ODBC ドライバーはありますか？

	まだ、ありません。Cees de Groot(C.deGroot@inter.NL.net)さんが作業
	しています。もし手伝ってもいいと思うようでしたら、彼に連絡して下さ
	い。

第４節：	Postgres95 の拡張についての質問
----------------------------------------------------------------------

4.1)	ユーザ定義ファンクション(関数)を書きましたが、psql の中から実行し
たらコアダンプをしてしまいました。

	問題は色々と考えられます。まず最初に、あなたのユーザ定義ファンクショ
	ンを単独のテスト・プログラムとして試してみて下さい。また、
	type_in() や type_out()ファンクションの実行中のように、フロントエ
	ンドがデータを待っている時に elog NOTICES を送っていないか確かめて
	下さい。
	[訳注：elog は error logging です。]


4.2)	次のようなメッセージを受け取ります。
	NOTICE:PortalHeapMemoryFree: 0x402251d0 not in alloc set!

	あなたは palloc されていない何かを pfree しようとしています。ユー
	ザ定義ファンクション(関数)を書くときに "libpq-fe.h" をインクルード
	しないようにして下さい。それをすると palloc が free のかわりに
	malloc になってしまいます。そして、バックエンドが保存領域を pfree
	しようとするときに、あなたはそのことを知らせるメッセージを受け取り
	ます。

	
4.3)	私は、Postgres95 のために粋な新しい型と関数を書きました。

	どうか、それらを他の postgres95 ユーザと分かちあって下さい。皆さん
	の行なった拡張をメーリング・リストに送ってください。そして、ゆくゆ
	くそうした拡張は contrib/ サブディレクトリに入ることになるでしょう。


4.4)	タップルを返す C ファンクション(関数)はどのように書きますか？

	これには極端な妙技を要します、あまりにも極端すぎて著者らは未だかつ
	てやってみたことがありませんが、原理的には可能です。手短に答えると...
	それは無理です。この機能は将来、やがて出来るようになります。

5.1)	バグの修正版はどこでみつかりますか？

	s2k-ftp.cs.berkeley.edu:/pub/postgres95/www/bugs というディレクト
	リにパッチと最新版があります。


5.2)	バグ・レポートはどういうふうにしますか？

	最初にあなたの見つけたバグが既に修正されたもの(質問5.1)ではないこ
	とを確認して下さい。そして、バグのテンプレート(雛型)ファイル
	(bug-template)に記して、それを
		postgres95@oozoo.vnet.net
	に送って下さい。

	[訳注：バグのテンプレート・ファイルは、bug.template と言う名前で配
	布に含まれています。]

	Postgres95 の公式なサポートはありませんので、バグ・レポートに対応
	してもらえるかも知れませんし、対応してもらえないかも知れません。
