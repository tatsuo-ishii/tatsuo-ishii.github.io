> とどのつまり、PostgreSQLの良いところってのは、オープンソースってことになっ
> ちゃいますね・・・（堂々巡りですが　笑）。
> でもやっぱりそれは、ソースが読めて、独自のPostgreSQLが作成できる程度の力
> があってこそ・・・ですね。そうでなければ、まったく意味が無いような・・・。

この話の元になっている部分について言えば、必ずしもそうではありません。

例えば、そちらで使っているシステムが PostgreSQL 6.5.3 を前提に
しており、そちらの使用範囲では 6.5.3 で重大な障害もないし、
機能的にも不足がないとします。

その場合、PostgreSQL が今後 7.0.2 から 7.1, 7.2, 7.3 … 8.0 … と
バージョンアップしていっても、6.5.3 を使い続ける事はできますし、
万が一 PostgreSQL プロジェクトが消滅しても使い続けられるわけです。

で、バージョンアップしたり消滅後に、そちらのサーバが増えたり、
支店や営業所が増えても、PostgreSQL 6.5.3 をインストールして
使う事ができます。

# サーバマシンが PC アーキテクチャマシンから SPARC マシンなどに
# なっても、その逆でも対応が可能という利点もあります。

商用製品では、多くの場合こうはいきません。少なくとも私の経験上では、
Microsoft 製品では無理です。:-)
# 実際には別に Microsoft 製品に限ったわけではないですけど。

6年前に MS Windows 3.1 + Access 2.0 でシステムを作成した。その
アプリケーションのままで何も不都合が無い。今度業務拡大に伴い
PC を増やしたい。と、いうような場合に、今さら Access 2.0 が欲しいと
言っても売ってくれません。

万が一 Microfsoft が消滅したり、ある製品系列のサポートを打ち切
ったり(これもどこでもよくある)されたら、バージョンアップ版でさえ
入手不可能になります。

こういう手配上の問題がある。というのが、おごちゃんが指摘した点の
根っ子の部分でしょう。
皆さんは手配上の問題に遭遇した事はあまり無いのでしょうか？

で、Access 2.0 から Access 95 にしようとすると、プログラムがその
ままでは動かないとか、再テストを強要される。 というのは Microsoft
製品のせいですが…

いずれにしろ、こちら側のシステムのバージョンアップや再テストが
ベンダー製品のバージョンアップタイミングによって強要されたり
するというのは、ベンダー製品を使う限り避けられない運命でも
あるわけです。
