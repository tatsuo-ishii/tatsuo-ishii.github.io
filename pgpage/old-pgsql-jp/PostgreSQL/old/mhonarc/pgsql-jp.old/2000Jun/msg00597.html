SAM ファイルしかなかった、COBOL プログラマーの伝統(笑)が、
脈脈といきづいているだけでは(爆笑)。

まじめな話、RDB で予備カラムを必要とするような用途は思いつき
ません。

何の為に予備カラムをとるのか、直接聞いてみた方が早いのでは？。

逆に、デバッグや検証用専用の列を入れておいて、デバッグ完了後
列削除するとかいうことはしたことがありますが。


> 皆様はテーブルを設計する場合にどのようなことを考慮して設計して
> おられるでしょうか？

オーソドックスな答えですが、いちど正規化してから、処理時間や
プログラム工数を勘案して、正規化を部分的に崩すという方法です。

崩しすぎると、保守性が悪くなるし、まじめに教科書通りの正規形では、
大変ということで、システムの目的や性格に応じてさじかげんという
ところです。

細かいことを言えば、NULL は、本当に必要な列以外は使わないよう
にする。default 0 NOT NULL を殆んどの数値型の列に付けています。
特に、エンドユーザーコンピューティングでは、NULL を理解しても
らうのは困難なので、使わないようにしています。

> または、特に「PostgreSQLの場合に限って」ということになると、
> どのようなことに気をつけていますか？

どのDBMでも同じですが、異なる DBM への移植性をどこまで考えるか
によります。システム設計書に、『ANSI/JIS SQL xx 規格 及び 他の
DBMとの互換性は一切考慮しません』と明記できれば、な〜んも考え
ないですみます。(^_^) 。

後はプログラム言語との相性かな。
C でプログラムする場合は、C 言語には、文字列型がないので、
DB の列、特にコード等は、できるだけ、integer になるように設計
するとか。
