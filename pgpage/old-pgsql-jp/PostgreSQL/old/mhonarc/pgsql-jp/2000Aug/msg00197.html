これだけだとposgresさんがそのデータが読めるという
こと以上も以下もわかりません。
yamadaさんはそれを読み書きする権利があるのですか？

>　　＜ここから質問なんですが＞
>　　　ODBC にて　Microsoft の　Access より　アクセスしたいのですが
>　　　どうしても出来ません。
>　
>　　　　☆☆☆　上記は　マシン　Ｂ　です　☆☆☆　
>　　　　　　　　OS は　Windows です　　　　　　

この件には直接関係無いですけど、Windowsって何種類もあるんで
そこまで書かないいけない場合もあるんですけどねそれほど意味無いです
Windows側の設定問題だった場合、ネットワークの設定方法は
NTと95/98で違うし。(2000は1度触った程度だから良く知らない)

>　　なお、Win から　telnet にて　Ａマシン　の postgreSQL　の
>　　操作は出来ています。

telnetでログインしたらソフトはWindowsの画面でも
中に見えるのはUNIXの画面だし、
postgresというユーザーで操作してると思うので
「ネットワークは正常に動いてます」程度の情報しかないですね。

>　　　--------------------------
>     createuser は　　yamada  で　LOCUS本のP.139 と同じです。

「本の通り」って質問は持ってない人は何もわからないので、
どういうコマンド打ちこんでどういう反応が返ってきたのかを
書いてください。
これ書いた上で「○○の何ページの操作」ってあれば
本持ってる人が入力ミスまで含めて指摘してくれるかもしれません


>　　   ここでチョット分からないのは、この時にパスワードの指定が
>　　　 無いのに、PostgresSQL Driver Setup  の中に
>　　　 Password 指定が　有ることです。

ohbaというアカウントがあって、createuserでohbaを登録した場合、
telnetしてpsqlだといらない(ログインで認証済)けど、
ODBC経由ではohbaのパスワードが必要です。

>　　　　　まさかこのパスワードとは　Linux に接続するための
>　　　　　パスワードではないですよね。

で、本持ってるから(それよりバージョンの古い本(PHP/FI)だけど)
「本と同じ」かどうか見ました。
本のバージョンでページも書き方も変わってるかと思いますけど
ざっと見たらこういう記述があります。意味自体に違いは無いはずです。

本の記述：
「山田社員のホストアカウント「yamada」と同じ名前で
　PostgreSQLアカウントを作成します」

Linux側にyamadaというユーザーを登録してますか？
yamadaでtelnetでログインできますか？
これをしてない/できない状態なら、Linuxに接続する為の
ユーザーアカウントが無いんですから「yamada」でODBC接続は
出来ません。

＃ここで、「まさか」とか聞いてるので、本の通りって言ってるけど
＃実際は本の通りにしてないってパターンじゃ無いですか？

>      情報不足とは思うんですが、何か手掛かりになるような
>　　　アドバイス頂けると助かります。

本の通りにするならその本を良く読んで下さい。
