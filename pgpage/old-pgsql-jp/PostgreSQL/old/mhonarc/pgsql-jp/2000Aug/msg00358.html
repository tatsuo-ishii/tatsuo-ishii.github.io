[pgsql-jp 16744] [Q]relation between index and update,delete speed

  • From: Yoshio Funaki <yoshio@apl.pref.akita.jp>
  • Date: Fri, 18 Aug 2000 08:58:09 +0900
お世話になっております。船木＠秋田県立図書館です。

以前このＭＬで、インデックスを作成しているテーブルに対して
updateやdeleteをかけると、インデックスを作成していない場合より
時間がかかるという報告が流れていたと記憶しているのですが
本当なのでしょうか？

と申しますのは、現在、FreeBSD4.0R+postgreSQL7.0.2(with MB)で
図書データベースを構築中なのですが、予約情報等１日数回定期的に更新が
あるものと、夜中に１日分まとめて更新するものと２通りの更新がかかるのですが
どちらにしろdeleteとupdateの固まりになるので、少しでも早く
作業が終わった方がよいのです。（あたりまえか　^^;）

日中の定期更新はデータ量も少なく、頻繁にアクセスされる可能性が少ない
テーブルなので、あまり気にしてませんが、夜中更新の分は、更新データ
テーブルともある程度の大きさなので、時間がかかることが予想されます。
＃　と言っても、クリティカルな業務ではないので全くＤＢを止められない
＃　訳ではないです。よって以前議論があった連続運用に関する問題は
＃　とりあえず考えなくてもよいと思います。

もし有意な差があるようでしたら
index drop -> update,delete -> vacuum analyze -> create index
のような作業手順にしようと考えているのですが、先達のご意見を
拝聴させてください。よろしくお願いいたします。
