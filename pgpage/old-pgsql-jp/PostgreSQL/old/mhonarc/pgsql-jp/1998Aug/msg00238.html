>>>>> On Thu, 27 Aug 1998 14:59:17 +0900, 川野 健一 <kawa@stgaia.co.jp> said:

> 　１．ＨＰ上で、ユーザに趣味を３０個の選択肢の中から、０〜３０個
> 　　　選択してもらい、その結果をpostgresに格納する。
> 　２．検索画面より、趣味を一つ選択し、該当ユーザを一覧表示させる。

あらかじめ趣味の数と種類が定まっているのであれば、DB テーブル上の
hobby1 カラムは将棋、hobby2 は読書とか定めておいて、登録アプリ
ケーションは将棋がチェックされていたら hobby1 に 1 をセットする
というように作る。
検索時に、将棋が趣味の人を探したければ、 テーブル名が test で、
ユーザ名のカラム名は id 、趣味をセットしてあるカラム名は上記の
通りだとして、

  select id from test where hobby1 = 1;

というような SQL を発行して検索する。ではダメですか？
後はレコード(行)をひとつづつ fetch して処理を行う。

データベースを使うのであれば、検索作業はできるだけ DBMS 側にやらせる
方がいいと思いますけど。
メモリと CPU の使用量は DB をただのファイルとして扱って、すべて自前の
処理で行うのとどっちが有利かはケースバイケースだったりするかも知れ
ませんが、アプリケーション的にはこっちの方が楽でしょう。

もっといい方法もあるでしょうけど、私だと上記のように愚直に
やる公算は 100% (^^;
