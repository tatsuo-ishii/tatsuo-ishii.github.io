  • Date: Thu, 28 Jun 2001 17:12:53 +0900
船木＠秋田県立図書館です。
現在公開している蔵書検索のDBが、とりあえずの作品で
いろいろ問題が出てきたので、基本的構造から見直しを
かけようとしていろいろtryしているのですが、現システムでの
一番の問題が、primary key設定していないため、同一データが
書き込まれてしまうものなので、それを解決しようとしているのですが
その流れの中で、外部参照をうまく使ってデータ更新を安全確実に
行いたいという目標があります。

で、具体的には
１）データについているuniq idのみのテーブルを作る
２）uniq idテーブルを外部参照する基本テーブルを作る
３）基本テーブルをinheritしたテーブルを作る（グループ用）
４）３で作ったテーブルをinheritした個別テーブルを作る（データを入力）

２）〜４）は、現在行っている方法の焼き直しです。
＃　同一カテゴリーのデータをあるテーブルからのinheritにすることにより
＃　select * from hoge_group* where uniq_id='hogehoge';
＃　でhoge_groupに属するすべてのテーブルをscanできてとても便利です。

で、期待していたのは１）のテーブルでuniq idを消去すると、それが伝搬して
４）のテーブルの該当タプルが消去されるということだったんですが、
実験したところそのようになりませんでした。
ちなみにPostgreSQL-7.1.2です。
＃　汎用から日時でデータを切り出す仕様なんですが、汎用の仕様により
＃　削除と更新・新規登録の判別が困難なので、飛んできたデータのuniq id該当
＃　データ（削除含む）を全削除->データ（削除含まず）を全登録という
＃　流れで行う必要があり、全テーブルからの削除漏れがあるとまずいんです

これは仕様なのでしょうか。
また、上記のことを実現するためには、各個別テーブルのuniq idを
直接uniq id テーブルを参照するようにしなければならないでしょうか。
先達のご教示をお願いいたします。
