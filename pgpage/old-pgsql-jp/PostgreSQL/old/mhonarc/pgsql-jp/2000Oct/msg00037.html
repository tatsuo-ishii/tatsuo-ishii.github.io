内田です。

企業内回線の場合、端末アクセス以外のデータも山と流れてるはすです。
SMTP とか PostgreSQL のデータとか :-)

もし、他のデータが暗号化せず素で流れているようなら、単に「暗号化」
という面だけで、「なぜ SSH ではなくTELNET」というのは無駄だと思いま
す。確かに Password が素で流れるのはナニでありますけど。

とはいっても、SSH/OpenSSH で ssh-agent/ssh-add 初めとした多くの便利
さを考えると、別に Security の問題が無くても、使っておいて損はないと
思いますよ。

それに「えーあそこの Host は回線が危ないから....」とか考えて、
Telnet/SSH 使い分けるのも面倒くさいし。

本家 SSH の場合のライセンス問題抜きにすれば、「SSH じゃダメ」の場面の
ほうが、「Telnet じゃダメ」の場合よりも少ないと思うので、Default を 
Telnet じゃなく SSH にするほうが何かとヨサゲに思えます。 

Ueta Masateru さんは書きました:
> ssh は licence の問題が、意外と severe です。したがって、企業では、お
>いそれと導入できない場合もあると思います。
>
>#"OpenSSH を使う" という選択肢は勿論ありますが。
>
>一般に、通信を暗号化するときの判断基準は、通信路が不特定多数から 
>access できるか否か、という点だと思います。専用線を使っていても、不特
>定多数の access を通す可能性がある場合(たとえば、Internet 網の一部を構
>成している場合)は、暗号化した方が適切でしょう。また、逆に Internet 経
>由でも、secure が保証されている VPN の上で通信するなら、無理に通信を暗
>号化する必要はないかもしれません。

そういえば、RMS いわく「SSH は FreeSoft では無い！！」ということで、emacs
の開発は SSH/CVS ではなく、RCS/kerberos ... (RCS のくだり自信無し) だそう
ですが  :p
