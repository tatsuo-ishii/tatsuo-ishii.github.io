SQLエラーにならないように事前チェックをしています。

SQL のエラーを利用するのは、デバッグの為と、
ＤＢ接続エラーや、トランザクションチェック位かな。

業務ロジックチェックは勿論ですが、データーベース
カタログを利用したりして、NOT NULL で、DEFULST 句
がない項目を NULL で INSERT しないかとか、NUMERIC
型をはじめ各データー型違反にならないかとか、参照
制約違反にならないかとか、全部チェックしてから、
ＤＢ更新処理(INSERT/UPDATE/DELETE)に入ります。

本当は、ＳＱＬの空実行みたいなことをして、実行した
場合に、制約エラーになる情報をエラーコードで取得
できると少し楽になるんですが。

# 十数年前の UNIFY 4.0 に、似たような機能があり
# ました。

でも、どうせ、業務ロジックでのチェックが沢山あるから、
SQL の制約条件のチェックが増えてもあまり気にならない
です。
