石井です。

> 　元質問の意味が、パラレル・クエリへの対応と言う意味だとすると
> 商用ＤＢＭＳ製品（Oracle、SQL Server など）は実装していますが
> PostgreSQL ではまだだと言う事になるんですよね？

パラレル・クエリは一つのSELECT文を複数のCPUで分担して高速化できる、と
いう機能なんですよね？だとするとPostgreSQLには実装されていません。ただ、
複数のユーザ(セッション)が同時にSQLを発行する環境では、PostgreSQLはセッ
ション毎にプロセスを作るので、このような場合にはPostgreSQLでもSMPの効
果がでてきます。

> # Oracle8i の場合は高価な Enterprise Edition に実装されてる。
> 
> > # Solaris for x86 だとどうなんだろ?
> 
> 　どっかで Solaris の方が WindowsNT Server よりも SMP の CPU
> 数増加にはリニアに対応できていると言うネタを読んだ記憶がある
> ので、
>        Linux(kernel 2.2)  <  WindowsNT Server < Solaris
> 
> と勝手に捉えています。Linux が Kernel 2.4 になり、Windows2000
> だとどうなるかは興味ありますね。

おー、そうですか。そうなると、是非とも 4 CPU の Solaris でテストしてみ
たいですね。

> # 個人的には IA-64 対応で、Windows2000 はずり落ちる気がする。
> 
> > ML でも繰り返し議論されている話題ですが、いつも「全てはシステムの設計
> > 次第」という結論に落ち着いている気がします。パフォーマンスを云々するた
> > めには、結局「ボトルネックになっている場所を明確にする」ことから始める
> > 必要がある訳ですが、その結果「ハードが原因」と結論付けられるのは、むし
> > ろ少数のような気がします。
> 
> 　結果的には、Oracle や Sybase などの商用ＤＢＭＳを使っても
> PostgreSQL を使っても、使ってる開発者の力量に依存するのが
> 現実だと考えます。
> 
> 　そういう意味では、PostgreSQL を利用する優秀なシステム構
> 築屋を増やすような動きがあると良いかもしれません。
> 
> # Oracle Master 対抗で、PostgreSQL Master 試験とか。。

うーん、そういうのは私はあまり興味ないです。あたりまえのRDBMSの常識と
か、OSに関する常識があれば、PostgreSQLではそこそこの性能を引き出せるは
ずだと思うので。
