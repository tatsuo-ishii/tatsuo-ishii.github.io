Enterprise を目指そうとすると、スレッドとか、主記憶の
有効利用とかいろいろありそうだけど、PostgreSQL は、違う
道を進んで欲しいと思う。

ANSI SQL 92 フルサポートは必須だけど、それ以上の高機能
を目指すよりも、使いやすさという方向を重視して欲しいと
思っています。
# Oracle 互換も必要かもしれない(^_^)。


> >  でも、Linuxは栄え、*BSDがイマイチであるというのは、「技術的に上なもの
> >が、必ずしもシェアと取るとは限らない」「技術的差異はユーザ数が増えるとい
> >つの間にか逆転する」ということを物語っています。で、そうなるためにはいろ
> >いろな意味での「使いやすさ」によるユーザ数確保と、「公開性」による技術フィー
> >ドバックが重要でしょう。
> 
> あと、ツールがいろいろな人たちによって開発されているのも大きいです。
> pgbashとか、pgacessとか・・

DB の開発ツールという面では、商用DB に比べて、
PostgreSQL は、見劣りするのは確です。

Oracle の Developers 2000 とか、Webdb とか、
Informix の 4GL その他各種開発ツールや、パッケージ
類の豊富さにはまだまだかないません。

# SAP/3 が、PostgreSQL に載ったらたいしたもん
# だけど、商売上やらないでしょうね。 

このあたりは、これからどんどん作る人が出てくる
だろうし、そういう方向に向って欲しいです。

ＤＢ本体の性能も大事だけど、実際の業務に使うた
めの、ツールやパッケージ重視にうつっていく時期
だと思います。

# オフコンが延びたのは、ソフトハウスがちょこっと
# 手を入れるだけで使えるパッケージソフトをメーカー
# 自ら豊富に用意したからです。この実用的な戦略は
# おおいに見習うべきかと。

> これの重要なことは自分の使いたいものは自分で作っちゃえ！
> という精神で使いたい人が勝手につくちゃって結果としてみんなが幸せに
> なれちゃったということです。
> (もしかしたら違うかも、でもそれはご容赦を・・・)

まさにその通りですね。

私も未熟ながら、PHP+PostgreSQL 開発ツールを企画中
です。GPLと独自のデュアルライセンスにしたいと思って
ます。

> まあ最近では Apach_PHP+PostgreSQL をレンタルサーバやホスティングサービス
> で導入していて初心者でも「いっちょやってやるか！」という人が増えている
> とか増えていないとか・・・

この傾向は嬉しいのですが、PostgreSQL について技術的
知識のないプロバイダーも多いようで、うわさ話ですが、
他の人の DB を自由に変更できてしまうとか、共同利用の
ためのガイドラインや注意事項がなにもないプロバイダー
もあるとか、いろいろ問題もあるようです。

> でも、この敷居の低さはOracleにはかんがえられませんよね！

１票！。

なんせ、オラクルは、Linux に載せるために、専用の
ディストリビューションを自ら作らなければならない
位だから。(逆にいえば、それだけ高度に OS を使いこ
なしているとも言えるのかな？？？)。


> >  「公開性」については、まだPostgresは青年期のソフトなんで、どんどんいろ
> >んな情報やレポートが流れ、改良がされ続けているので、商用ソフトなぞ目じゃ
> >ないでしょう。
> >
> >  さて、果してOracleはいつまで、その優位を保つことが出来るでしょうか？

1998年の秋でしたか、オラクル副社長の話として、将来いつの
日か、Oracle より優れたフリーソフトのＤＢが出てくるかも
しれない。でも、オラクルは、サポートを商売とするから困ら
ないというような話が日経バイトかなにかに載ってましたね。

ただ、Enterprise 版といった、単価の高い層を主要顧客と
する商用ＤＢと、PostgreSQL とでは、当分一部競合しても
棲み分けする時代が続くように思いますが。
