ということなんでしょう。ただ、SUGIYAMA さんに勘違いして欲しくないのは、
Web ページに記載されている「検索もできる」って事と、後藤さんのおっしゃる
where 句の対象とはなり得なくなる・・というのは状況が違うということです。

Web ページに書いてあるのは 受け取った SQL 内の対象部分 (where 条件の値)に
対して、暗号化処理を行なって、その値で書き直した SQL を PostgreSQL に送る
ような中間層があれば、可能です。と言ってるのです。当然後藤さんもこんなこ
とは承知の上で where 句の対象になり得ない とおっしゃってる(と思う:笑)ので
すが、なんせ、検索を可能にするためには統一キーワードで暗号化しなくてはな
りませんから、
> そんな事を気にするくらいなのであれば、crypt (3) を使ってデータを暗号化
> するアプリケーションを書いた方がいいんじゃないですか？
という結論になるわけです。

最終的には、Webサーバー - DBサーバー間の暗号化ならいざ知らず、「DB 内に暗
号化されたデータを Insert しても良いって言うのなら、不安に思う暗号化アル
ゴリズムは使わなきゃいーじゃん？」ってことですね。
案外、ビットシフトさせるだけのプログラムとかの方が、今日びのクラッカーに
は効果あったりして・・（ない？汗）
