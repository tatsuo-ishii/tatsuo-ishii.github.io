　すこし整理して私見を述べます。

　まず、最初に少し出ていた話で、列の増減で対処するアプロー
チが「リレーショナル」だといえるかという点ですが、厳密にい
えばレギュラーな処理が集合演算だけで処理できないことになる
のでリレーショナルなアプローチとはいえないのかもしれません。
　しかし、リレーショナルの定義に当てはまるかどうかだけでは
適切な判断はできません。
　前にも書いたように、RDBMS を利用するのはそれが便利だから
であってリレーショナルは絶対善だからではありません。したがっ
て、どのようなアプローチが適切かは、現実にどのようにすれば
便利で不都合が少ないかから決定しなければなりません。

　最初からきっちりとテーブルを作り込むのが適切な場合がある
（さらに多くの場合それが適切である）ことは否定しないのですが、
今回 web で公開されていたシステムのように、そもそも列が増減す
るのが最初から予定されている場合には、各所のクライアントも始
めからシステムテーブルを見に行くように作り込めるのですから、
それによって修正の手間云々の問題は生ぜず、むしろ、通常の方法
でデータを扱えるメリットのほうが大きいと考えられます。
　なお、各 DBMS で列を削除するときにトランザクションや制約の
処理がうまくいかない実装もあるかもしれませんが、その場合はも
ちろんこの方法では駄目なわけで、その辺りは前提としてうまくい
くと仮定した場合の話です。postgres ではそもそも列の削除ができ
ないので元々仮定の話だったことを想起してください。

>> 　また本質的な問題は、added_item テーブルを使う方式では型や
>> （参照）制約など DB を使う上で重要な機能が使いづらくなるとい
>> う点でしょう。これらは工夫すればまったくできないことではない
>> のですが、そもそも何のために DB を使うのかわからないような設
>> 計となったのでは意味がありません。システムが提供しているサー
>> ビスがある場合にそれを使って楽をできるというのがシステムの意
>> 味のはずです。
>
>私の考えでは、これは何ら本質的な問題ではないと思います。
>データベースのレベルでの各種チェックに依存したシステムでは、
>多くの場合問題があります。
>＃速度とか、メッセージとかいろいろ。
>通常は、RDBに食わせる前に、エラーチェックなりなんなりをすることに
>なります。

　型や制約等によるチェックはクライアントとDBMSの両方でやるべき
ではないかと思うのですがどうでしょうか。また型が合っていないと
やはり後々手間が増えそうに思われます。
