石川@名古屋と申します。

小型ROM化LinuxのML(si-linux)で今、小さな DBを話題にしていますが、
postgreSQL は小規模な DBでもオーバーヘッドが少なく構築できるでしょうか？
という質問をさせて下さい。

想定するDBの規模は、1レコード 256-1024byte、 全 1000-10000レコード程度です。
(syslog のようなものを格納するのだと思って下さい。それにしては 1レコードが
長いではないかというつっこみはご勘弁ください。)
テーブルは一つです。(正規化は不要と想定しています。)

このDB を載せるLinuxマシンは小型ワンボードマイコンです。
ディスク(コンパクトフラッシュを使用します。)、メモリの制約が厳しいので
(共に数十MB程度)、DBは小さければ小さいほど好ましいです。

今、DBの候補としては 1)テキストファイル 2)MySQL 3)postgreSQL が挙がっています。

なぜ DB 化したいか? SQL で検索したいからです。
(grep , awk, perl 類ではなく...ただし、SQL 構文が使える script でも可なので
1) テキストファイル が候補に挙がっています。)
検索のための select, 最低限の保守のために insert, delete 位しか使わない
と思います。(一定期間経過した古いレコードは消去する。)
index はもちろん使いません。

その他、小型化優先なので、可用性や保守性は犠牲にしても構わないです。
つまり、すこしくらいDBを止めるのはありだし、保守しにくくても OKです。

postgreSQL の Administrators Guide によれば、
インストール後は、
 - /usr/local/pgsql以下、 2MB 占有
 - 空DBとして 2MB 占有。flat file にDBデータ保存する場合はその5倍必要。
と書いてありました。
(インストール時にはもう少し、資源を占有しますが、インストール時のリソースは
何とかなるのでスリム化の考慮対象外です。)

上記規模の DB を格納すると postgreSQL はどのくらいのサイズになるでしょうか？
また、postgreSQL インストール時、や DB 構築時に小規模DBを考慮した設定テクニック
などがあるでしょうか。
またはpostgreSQL のサブセット化した構築はできるのでしょうか？

なにかご意見、ご示唆いただければ幸いです。
よろしくお願いします。
