> > > うーむ、他のカラムではダブってしまう可能性があるから、自動的に「一意」
> > > にするためにserial型を普通は利用するのですよね・・・????
> > > 「serial以外の部分で一意であれば・・・」では、本末転倒な物になってしま
> > > わないでしょうか??
> >
> >   でも原理的に考えてみれば、カラム内容が完全に同じ行が複数あった場合でも
> > 「今INSERTした行はこれだ！」と判別するのは容易ではないと思います。
>
> 完全には無理でもかなりのレベルまでは可能でしょう。
> CREATE TABLE test (id serial primary key,txt text);
> を例にプライマリーキーがserial型の場合。
> select max(id) from test;
> まずこのようにして、プライマリキーマックスを読み込んでおきます。
> insert into test (txt) values ('ABC');
> を実行後
> select id,txt from test where id > マックスの値 and txt='ABC';
> で抽出するのです。

  これではプライマリキーの自動発番の特徴を知っている必要があります。

> タイミングの問題でダブル可能性もありますが、ODBCでの利用は同施設内のLANな
> どで利用する場合が多いと思うので、実際はそこまで重なる事はまず無いと思いま
> す。
> 現在の方式を優先にするか、この方式を優先にするかオプションチェックで選択で
> きれば良いかもしれないと思うのですが、いかがなものでしょうか？

  前のメールにもちょっと書きましたが、この処理をしているのはMS-Jetエンジンで
す。ですから上記チェックボックスを付けるとしたらMSに頼むことになってしまいま
す（笑）。

  なお、仮にODBCドライバ内で同様な処理をするなら、単にoidを使えば良いだけで
す。そうすればダブル心配もなくて確実です。
