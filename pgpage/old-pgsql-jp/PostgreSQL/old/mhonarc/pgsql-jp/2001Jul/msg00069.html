こんにちは。藤本です。
いつも参考にさせていただいております。

PostgreSQL 7.1.x から view に関する制限が緩和されたのを機に、
シーラカンス本第３版を参考にしながら、6.5.3 → 7.1.1 への移
行を試行している中で、表題の件でご教授いただきたく、お便り申
し上げます。


現在、11個のテーブルを結合する view を作成してその view に対
して様々な条件で検索、ソートを行うことを試行しております。

この時、一般の table に適切なインデックスを設けることで検索
の高速化を図るように、view に対しても同様の工夫が可能なので
しょうか？。

それとも、複数の table を結合した view に対する検索は、

　・その view の定義を実行する時間
＋
　・view に対する検索を実行する時間

が必要で、その“和”は、

　・結合せずに手動で各テーブルに検索を行う時間の総和

とほぼ変わらないのでしょうか？。

view を利用することで、よく使う複雑な問い合わせを view に定義
しておけるという有用性はわかるのですが、速度面の方はどうなの
かご存知ありませんでしょうか？。

わかり難い質問かも知れませんが、よろしくお願いいたします。
