>> 設計によります。PostgreSQL では、テーブル単位でロックをかけるので、一
>> つのテーブルに書き込みが集中し、かつ処理に時間がかかるようだと応答が悪
>> くなります。
>
>なるほど。レコードはなくてテーブル単位なのですね。同一テーブルに書き込み
>が多いと確かに問題になりそうですね。
>でも、普通は検索やブラウズは多数でも書き込み処理が集中することは少なそう
>なので、何とかなりそうです。

そうですね。PostgreSQL の場合、read は shared lock になるので、ただ検
索するだけなら複数ユーザが同時にアクセスできます。ただ、同時ユーザ数が
ちょっと少ない(デフォルトで32)なので、その辺が問題になるかも知れません。
その場合は、コンパイルし直しでユーザ数を増やすことができますが、同時に 
shared memory を広げたりする必要が出て来ると思います。

>漠然とした質問で失礼いたしました。実は、自分でも具体的な基準がなかった
>のです。例えば、P2 333にメモリ128mbという現在ハイスペックといわれるマシ

>ね。
