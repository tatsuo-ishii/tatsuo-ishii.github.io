フォローがない事が「PostgreSQL で電子出席簿のような公文書を管理するのが
危ない」と、どのようにつながるのかがよくわかりません。^^;

> 	それとも、自分で苦労して探しなさい。あるいは、Access の help だけで
> 出来るようになるということかな?

私自身、Access や ODBC ドライバに詳しくはないのですが、一般に、ある
DBMS を ODBC 経由で使った場合、その DBMS の全機能を使えない事になる
ケースもあるようです。あるいはこれは使い方の問題なのかも知れません。

しかし、ここは PostgreSQL の ML なので、Access + ODBC の使い方を聞く
のは本来の意味で言えば場所が違います。
PostgreSQL との組み合わせで何か特異な現象が起こるのであればその限り
ではないでしょうが… まあ、そこまで厳密な事は言わなくても良いのでは
なかろうかと個人的には思いますけど。
# しかし、Access の話しが延々と続くようだと困るのも確かです。^^;

Access 本体や Access + ODBC については、市販書籍が溢れるほど出回って
いるのですから、それらをあたって見た方が、内山さんが得るところは多い
のではないでしょうか。

> 	石井様、Gotoh Kazumasa 様。私が、Postgres をけなすつもりで、元記事
> を書いたわけではないことはお分かり頂けたと思います。電子出席簿を PostgreSQL
> で実現するには、心配なことがたくさんあって二の足を踏んでしまうという不安感
> から記事を書いてしまったわけでした。どうもすいません。

私も PostgreSQL に詳しいわけではありませんが、「心配な事」があるので
あれば、「こういう面は PostgreSQL ではどのような扱いになっているのだ
ろうか」などと、もっと具体的な例をあげた方がいいと思います。

ただし、ここはサポート ML ではなく、ユーザーズ ML なので必ずしも内山さん
が期待するような答えが返ってくるとは限りませんけど…

> 	その他に、セキュリティも気になっています。6.2.1 までだと、IP アドレ
> スでアクセス制御をしていたようにおもいますが、学生さんも同じネットワークを
> 使っていて、IP アドレスの捏造があるかもしれない。DB-username と password
> 認証でデータベースアクセス制限をかけて、grant or revoke でテーブルアクセス
> 制限をかけなきゃいけないのかなとか考えています。

ユーザ認証の面については、私も興味を持っています。この部分は
6.3 以降では強化(改善)されたようにも聞いていますが…

ただ私の場合は、内山さんのように Access + ODBC 経由でのアクセスでは
なくて、Apach + PHP/FI との連携による使用を考えています。

ユーザ認証を行なって、参照権限と更新権限がうまく切り分けられればと
思いますが、まだ試してないのでよくわかりません。^^;
