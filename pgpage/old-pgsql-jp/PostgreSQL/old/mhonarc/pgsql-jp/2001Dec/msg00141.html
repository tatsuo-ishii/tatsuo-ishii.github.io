最近 Servlet から離れているので、はずしているかも知れませんが…

From: Dai NAKAMURA <dnimail@sanix.co.jp>
Date: Sun, 09 Dec 2001 16:28:50 +0900

> もしかしたら、PostgreSQLではなくアプリケーションで解決すべき問題
> なのかもしれないのですが。

基本的にはこっちの方で解決すべき問題であろうと考えます。

> JDBCなどで、postgreSQLにサーブレットなどから接続セッションを確立し
> て、SQL文を実行したとします。
> このとき、たとえば端末側の事故や、CPU負荷によるレスポンスの零下など
> の理由で、ブラウザが異常終了した(もしくはユーザーによってされた)
> 状況などが発生した場合、残された接続セッションのプロセスを強制的に
> 落とすことはできるでしょうか？

ここで言っているのは PostgreSQL のセッションですか？
つまり、postgres プロセスがずっと残っているとか？

いずれにしろ、Servlet API のバージョンにも依存する部分もあるかも
知れませんが、基本的な方針としては当該オブジェクトが破棄される
タイミングで呼び出されるメソッドの実装を行い、そこで必要に応じた
後処理を行えば良いのではないのでしょうか。

それとも Servlet のスレッドもそのまま残っているのでしょうか？
その場合は Servlet エンジンの設定や、Servlet プログラムそのものに
問題があると考えます。

> 当方の環境は、
> RedHat 7.1+PostgreSQL 6.5+JDK1.3
> です。

Servlet の話が絡むのであれば、何という Servlet エンジンを使用している
のかという事とそのバージョン、そして Servlet API のバージョンを明記
すべきでしょうね。

私自身は Servlet まわりは忘れている/最近の事情に疎い部分も多いですが、
詳しい人にとっても JDK のバージョンだけ書かれても意味がないです。
