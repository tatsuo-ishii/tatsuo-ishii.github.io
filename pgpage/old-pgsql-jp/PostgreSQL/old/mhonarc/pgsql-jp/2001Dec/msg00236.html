石井です．

> 現在の Oralce の DATE 型を SQL 標準の DATE と TIMESTAMP にするのは
> ユーザに対するインパクトがあり過ぎて、今となっては無理なような気が
> します。
> NUMBER と VARCHAR2 に関しては… どうかな？
> 
> Outer Join など SQL の記法については、ORACLE 独自と SQL 標準の
> どちらも通るようにしてくれよ。とは思います。
> 
> 実は通ったりするのかな？ 未確認です。 (^^;

JOIN構文は無理だと思いますが，INTEGERやVARCHARは受け付けると聞いたこと
があります．であれば，なおさらVARCHAR2をOracleが推奨するのか不可解です．
「過去の資産の継承」が理由なのであれば，これから新たにOracleを導入する
ユーザにVARCAHR2を推奨する理由はないはずです．

また，本当に「過去の資産の継承」を言うのであれば，昔のOracleのバージョ
ンもサポートすべきだと思います．でも実際にはしませんよね．

まあ，このあたりビジネス的に無理なのは理解できるので，過去のバージョン
がサポートされないのをけしからんと言っているわけではないです．ただ，ベ
ンダーが良く口にする「過去の資産の継承」というのは，本当に顧客本位かと
いうとそうではなくて，単にビジネス上売り上げに向上につながるならやりま
す，という文脈が常に背後にあることを分かっていた方がよいというだけです．
このあたりがOracleに限らず営利企業の限界なわけですね．
# というか，クローズドソースのビジネスモデルの限界と言うべきか．

逆に，そういうしがらみとは無縁に，長期的にユーザの利益になる方向を追求
できるのがフリーソフトの良いところではないかと思います．
# もちろんソースがあれば，その気になればユーザ自身でメインテナンスしな
# がら過去のバージョンでも使っていけるということもあります．

過日ストールマンが「フリーソフトウェアの目的は多くの人に使われることで
はない。使う人を自由にすることだ」と述べたそうですが，これは非常に含蓄
のある言葉だと思います．
