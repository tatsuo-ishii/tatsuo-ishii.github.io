まつむらです。外している可能性大です。

> に以下のセキュリティ例外が発生してコネクションをはることができません。
(ばっさり略)
推測ですが、この手の exception は $CLASSPATH に起因すること多いです。
netscape を立ち上げた時の $CLASSPATH に問題のクラスが含まれていると
そっちを netscape 君は優先し、かつ、ローカルからロードしたもんとして
扱います。んで、そのクラスから別のサーバ (物理的に同じ云々ではなく、
FQDN 的に違うの意味) にコネクションを張ろうとすると上記の exception
が発生すると。

標準の applet セキュリティモデルでは、ダウンロードしたホストとしか
tcp コネクションは張れないことになっているです。

どこからクラスファイルを持ってきたかを見るには、netscape オリジナル
の jvm だと、java console を開いた状態で、g のキーを押すとデバッグモ
ードになるので、目で確かめるか、httpd のアクセスログで確認するです。

んで、対処は、
1) unsetenv CLASSPATH ; netscape して、意地でもネットワーク越しに
   ダウンロードしてもらう。この場合、postgres.jar とかも httpd か
   ら見えるところに置いておく必要ありです。(老婆心ですね)

2) もしくは、署名付きの applet をでっちあげるかです。
   このやり方は、jvm 毎にまちまちです。
   netscape オリジナルの場合は、先月号の java magazine に書いて
   ありました。java world だっけ？
   activator の場合は、その配布元の近辺にやり方はあったと思います。

ご参考になれば。
続くようでしたら、java-house のほうがベターな気もしますね。
