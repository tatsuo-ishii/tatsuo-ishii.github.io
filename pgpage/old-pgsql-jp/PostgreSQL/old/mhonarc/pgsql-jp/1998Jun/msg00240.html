私はごく普通に cvs を使っています。cvs の良い点は、

o ファイル単体ではなく、ソース全体のバージョンを管理できる。
o バージョンの枝別れを作ることができる
o バージョン間で自由に差分を取り出せる。すなわちパッチを作ることができ
  る。

あたりですね。たとえば、私は 6.3.2 用の MB パッチを作っていますが、
MB1, MB2, MB3 とそれぞれバージョンを定義し、

o MB3 パッチ→オリジナルと MB3 の差分
o MB2-MB3 パッチ→ MB2 と MB3 の差分

のようにしてパッチを作っています。当然ですが、cvs はバージョン名(MB3 
とか)を単なるシンボルとしてしか扱いませんので、バージョン名に対する意
味付は利用者がドキュメントなどの形で管理する必要があります。これをしっ
かりやらないと、すぐにわけが分からなくなります:-)最近のパッチで、"set
query_limit patch" がありましたが、以下のようなメモが残っています。
----------------------------------------------------------------------
cvs import -ko -m "6.3.2" postgresql-6.3.2 postgresql V6_3_2

# "set querylimit to" patch 用のブランチタグ作成

cvs rtag -b -r V6_3_2 QLIMIT postgresql-6.3.2

# work file を co
cvs co -r QLIMIT postgresql-6.3.2/src

# patch の作成
cvs rdiff -r V6_3_2 -r QLIMIT postgresql-6.3.2 > ~/tmp/qlimit.patch

---
