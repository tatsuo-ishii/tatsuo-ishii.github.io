池田です。

> > 　・decode は case に書き換える、case の方が判り易い。

　ちなみに、decode は Oracle 独自の if 的なモノ。case の方が
SQL標準です。decode は、ほとんどパズルです（苦笑）。

> > 　・NUMBER 型は存在しないので適切な数値項目にする

　Oracle の場合には int や long も内部は NUMBER のエイリアス
だったはず。NUMBER型は有効桁が３８桁で、これにあたるのは
PostgreSQL だとNUMERIC型（１０００桁）だと捉えてます。
　が、Oracle の NUMBERは索引張れるんですが、PostgreSQL
で索引が張れる有効桁数が多い数値項目ってないのかな？

> > 　・MVCCの考え方は Oracle と同様だがさらに rbs の大きさに
> > 　　依存しないので便利

　Oracle の場合には、トランザクションが終わるごとに vacuum
されているイメージに近いですね。ですから、読み取りしている側
に旧データが途中で読めないエラーが戻ることもある。

> かなり興味深いです(非常に参考になります)。

　オラクルも昔に比べると情報が増えているし、技術者も増えて
いるので、それとの比較が役に立つなら整備するのも良い？

> 実は昔、「Oracle のすごいところ」というような記事で、ファントムを発生させ
> ずに更新トランザクション中のレコードを読み取ることが出来る・・というよう
> なのがあったのですが、これと MVCC の違いに興味がありました。
> MVCC ではファントムが発生するのだろうか？と不安を持ちつつも実験せずに今日
> にいたっているのですが・・・。

　うーん、MVCC の実装方法によってはファントムは発生して
しまうと考えます。
　Oracle は、、、昔に調べたのを忘れてます（爆）。

　実験としては、３つのクライアントを使って（SQL*Plus、pgsql）
検索と更新と挿入をしてみれば良いのですが。。

> このあたりを詳しく知りたいのですが、多々ある比較表(他の RDBMS も含む)で、
> ここまで言及しているものは見つかりませんでした。
> (もしあれば教えてください)。

　知る限りでは、単純な比較表では丸バツが多いのでたぶん
役に立たないです。
　役に立つのは、「Oracle から SQL Server への移行」とか
「SQL Server から Oracle への移行」などの技術資料などが
良いでしょう。日本オラクル、ＭＳＫＫのサイトを漁ると出てくる
と思います。

　これに PostgreSQL の情報を重ねて考える、ってのが良い
と捉えます。とは言っても悩んでますが（苦笑）。

ではでは。
