Web + DBMS という前提だったので、『pgsql-jp では不適切』
とおっしゃられたのだと思います。

PostgreSQL が、商用ＤＢと比べて一番見劣りするのは、
帳票印刷とか、業務パッケージとか開発まわりの細々した
ツール類です。

PostgreSQLが商用ＤＢと同様に使われはじめている現在、
帳票印刷といったＤＢまわりの話題も pgsql-jp ML で
どんどん取上げていって、オープンソースのツール類が
あちこちで開発・公開されるきっかけになればいいと、
私は、思っています。この手の話が多すぎて邪魔にな
るようなら、別 ML に分ければいいし。

------------
磯川さんの質問に戻ると、PHP で、伝票印刷など書式
をきちんとするなら、市村さんのおっしゃるとおり、
PDF でしょうね。詳細は、PHP-jp ML の過去ログをご覧
になるといいでしょう。
検索:
http://search.net-newbie.com/cgi-bin/namazu.cgi

雑誌では、技術評論社 WEB+DB 誌 Vol.1 2000 の P.45 に、
数行印字する簡単な例が載っています。

宅急便伝票など感圧紙用のインパクトプリンターだと
特定のプリンター用の印字データーをサーバー側で
作成してダウンロードしてからプリントとか、WAN/LAN
環境なら、サーバーから直接 LAN プリンターに出力
なんてやってます。

-----------
商用ＤＢの Informix から、PostgreSQL に移行して
一番不便なのが、簡易入力ツールや、レポーターがない
ことです。Informix の場合は、テキストベースですが、
SACEGO というレポーターがあって、CONTROL BREAK で、
小計、中計、大計といった合計処理や見出し処理、
COLUMN 位置指定による位置決めなどが簡単にできて、
エンドユーザーでも少し勉強すればプログラム可能で
した。PostgreSQL だと、PHP とか、Perl や、JAVA を
使うことになるんでしょうが、CONTROL BREAK 処理等
をエンドユーザーにプログラムさせるのは一寸難しい
ですね。今更、COBOL や RPG(ゲームじゃないよ(笑))
でもないですしね。

オープンソースのレポーターとか帳票作成ツールがで
てきて欲しいものです。
