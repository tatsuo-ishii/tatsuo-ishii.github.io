北村＠アシェット さん、回答ありがとうございます。


> 配列は鬼門と呼ばれている(勝手に呼んでるだけ？)ので、なるべくならテーブル
> を2つに分けて(正規化して)設計した方が良いです。

なんか本当に鬼門なんですよね。パフォーマンス上も配列は避けた方がよさそうで
すし、たかが中身を検索するだけでも contrib/array を別途組み込まなければなら
ないとか...

> 要は、配列要素を縦列に吐き出せれば良いだけで、PL/pgSQL を使えば配列要素が
> n でも OK なものを作成できると思います。C 関数でも良いですし。

C言語自体は苦にしないのですが、PostgreSQLの関数は書いたことがないですし、自
分で書くとパフォーマンスとか信頼性や開発工数の問題が発生するため、できるだけ
避けたく思います。

> 今回、配列要素を2つと限定すると、こんな方法があります。
> create function fGetKey2(text) returns setof text as '
> ...中略...
> CREATE

ところが配列要素数が可変長なので、配列要素数をきちんと調べてから上記と同様な
ことをやらなければなりません。
そこで、自分としては以下のスタンスで進めようかと思います。要素数が可変長の配
列を縦に出力する関数が既に開発されており、評価も終わっていて、無償で使用させ
ていただけるのであれば、それを使わせていただく。
もしもこのような関数がないのであれば、テーブルを設計し直し配列の部分を別の
テーブルに切り離す。

以上、何かよいリソースがありましたら、教えてください。
