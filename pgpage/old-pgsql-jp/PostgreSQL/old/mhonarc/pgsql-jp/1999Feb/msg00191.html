松江高専の内山と申します。

Tatsuo Ishii さんは書きました:

>6.5 では MVCC (Multi-Versioning Concurrency Control(?))を使います。こ
>れは DELETE や UPDATE で行を更新しても、以前の行の内容が保存されるシス
>テムです。これと各行が持つトランザクションIDを併用することにより、並行
>して走るトランザクションがお互いに干渉することがなくなります。これは、
>基本的にはロックが不要であることを意味します。実際、6.5 では、違う行を
>アクセスする限り、テーブル全体にロックを掛けることなしに、複数のトラン
>ザクションが同じテーブルを同時に更新することができます(今は、どちらか
>のトランザクションが commit するまで待たされる)。

	研究所にいたころは、ここいらの勉強をして特許とか書いてましたことを
思い出しました。
	今、急いで教科書を読み直したところ、全体として石井さんの行っていることに
間違いはないと思います。補足すると、

	推測するに PostgreSQL は単なる client-server 型のデータベースから脱皮
しようとしてる。目指すところは distrebuted (object) relational DBMS ではないだろうか。
それなので、Concurrency Control の概念を導入しようとしてると考えられる。
	トランザクション管理に　Concurrency Control を取り入れると、lock という概念より
は、transaction manager という概念が重要になってくる。すなわち、トランザクションを管理する
プロセスが各トランザクションを監視して、一貫性・整合性を保つ。
	これは distributed DBMS だけではなく、distributed file system など OS レベルでも
用いられる考え方である。
	今のところ、このくらいしか思い出していませんが。参考書としては

	edited by dr. Sape Mullender, Distributed Systems, Addison-Wesley Pulishing Company

これは英語です。現在、手元にはこの本しかありません。日本語で良い参考書は

	電通大におられる（おられた）：前川先生の書かれた分散システムについての参考書
	があったはずです。購入を予定しています。

しかし、この辺は　もう十年くらい前にモデルが確立できているので、かなり枯れた技術と
思われます。ですので、本家 PostgreSQL の人たちも、単に参考書に書いてあるモデルに
従うだけではなく、より良いトランザクション管理のモデルを模索中ではないかと思われます。
	PostgreSQL が　また一段と魅力を増しますね。ついに　私も高専 4 年生の
データベース概論１の講義の時間中に SQL の演習に PostgreSQL ver. 6.2.1 を直に学生に体験
させることができました。自作のテキストには　reference として石井さんのページの URL を
載せておきました。もしかしたら、来年度より松江高専の各研究室で PostgreSQL を使い始めるか
もしれません。今までは、うちの研究室の特許だったんですがね。かわいい教え子のためには
仕方ありません。去年担任をしていた子達です。
	その際に、学生が当ML に入会するかと思いますが、なにとぞよろしくお願いします。
