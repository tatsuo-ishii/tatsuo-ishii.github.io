> う〜む、ちょっと疑問ですね。
> まず、postgresql.jar への CLASSPATH が PostgreSQL 6.3 or 6.4 の
> postgresql.jar へ向いていないか御確認下さい。
> # 違ってたらごめんなさい。

6.4.2 の環境は消してしまったので、それはありません。
ただ、矢内原さんもおっしゃっているように、私の使い方では問題が
起きないというだけのようです。(^^;

> 通常 Java プログラムはプラットホームに依存しませんが、
> 日本語コードにも依存しない設計をするためには、入力されるコード
> を判定して Unicode にエンコーディングしなければなりません。
> 私の場合
>         byte b[] = nihongo.getBytes("8859_1");
>         nihongo = new String(b, "JISAutoDetect");
> などとしています。

Web ブラウザ経由などの場合、<FORM> で送られてくるものにはこのような
指定をして UNICODE に変換しています。

ロケールの問題だという事だと思いますが、私の使い方… DB が EUC で
FreeBSD 上での Java アプリケーションや Servlet では、LANG=ja_JP.EUC
ならば ResultSet.getString() した時の結果が UNICODE で帰って来ていますし、
UNICODE 文字列を executeUpdate() すると EUC で DB に入っています。

> # Windows なんかは LANG なんて概念はないでしょ。
> # また Servlet の場合を考えてみて下さい。ApacheJServ
> # を使うとしましょう。jserv は apache によってロードされ
> # jserv が java 或いは jre を使って class ファイルを実行
> # します。この場合 LANG はどうします？

JServ では、jserv.propaties で環境変数の指定を行なう事ができるので、
とりあえずそれを使って LANG を指定しています。

結論としては、日本語などの文字コードを扱ってもおかしくならないように
6.5 用の JDBC ドライバのパッチが出てくるまで根本的な解決にならないし、
最終的にはそれが本家のソースにマージされるようにならないと、この手の
問題がリリースのたびに起こるという事ですね。お騒がせしました。

ところで、おぎそさんの方では私が別の mail に書いたような、6.5 の
JDBC ドライバでホスト名やポート番号を指定するとうまく動かないという
現象は起きておりませんでしょうか？
