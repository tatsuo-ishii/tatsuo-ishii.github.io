なんか、みやかわ＠ホビー・データ様と、
話が食い違ってるなと、思ってったんですが、
ようやくわかった気がします。

確かに、定数で持っている方が、
計算量が少なくなりますね。

ただ、メンテナンスをするのが、
DB + フロントエンド(私の場合 PHP) の定数。

で、動的なチェックの場合、
計算量は多くなりますが、
メンテナンスは DB のみ。

みやかわ様は、計算量を少なくし、
処理を高速に終了させる事を優先させている様子。


で、一方、怠け者な私は、
メンテナンスするのが一つだけで良いなら、
そっちをとったということですね。

一つのテーブルで一つの VARCHAR field なら、
私でも、メンテナンスしようかなと思うのですが、
複数のテーブルで、それぞれ複数個の VARCHAR があると、
もう嫌です。
# 様々な仕事に追われる私には、
# メンテナンスのコストが払えません。
# 職種は PG なんですけど、明日は屋根にアンテナを設置に出張です。


で、例で出していたテーブルは単体で、
しかも VARCHAR が一つだけだったので、
こちらの状況は伝えられなかったように思います。
# エスパーな技能が必要だった・・・。


で、不思議に思ったのが
> # それこそ、INSERTした後、たしかにINSERTに成功したかどうかSELECT
> # して確認するような態度じゃないです？

USER 様が入力を行った後、登録ボタンを押していただくのですが、
登録内容を反映したものを画面に、表示しています。

USER 様に入力に間違いがないか、確認していただいてますけど、
そういう INSERT/UPDATE 後に SELECT 結果を表示って、
しないものですか？



> /*--------------------------------------------------------------*/
(snip)
この文章は、そう思いますが、
メンテナンスのコストが払えない私には、
動的にチェックを行う方が良かったのです。
