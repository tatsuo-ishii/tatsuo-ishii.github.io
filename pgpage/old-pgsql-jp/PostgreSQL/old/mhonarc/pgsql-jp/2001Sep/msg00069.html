| > 　もちろんINSERTする前に文字列の長さ等はチェックしておくべきですが、
| > 僕が思うに、そういうのはSQLを生成するアプリケーション側でやる作業

|
| 私もアプリケーション(PHP)側でやろうと思っています。

　INSERT/UPDATE する際に、アプリケーションで行うか DB の CHECK 制約に頼るか
はシステムの設計方針もあるでしょうね。
　文字列長くらいだったら、わたし自身は DB に SQL を投げて、その結果が CHECK
制約に引っかかったか調べるより、アプリ側で先に確認した方が楽な気がします(気
にしなければならない事が減るし)。

　で、判らないのがこの記述(↓)
| 作業の最中に、文字列がいくつ入るか知っているのは
| PostgresSQL になると思います。

　「文字列がいくつ入るか」は「文字列長がいくつまで入るか」だと解釈します。
　「作業の最中」ってのが判りませんけれど、開発中という意味で捉えるならば、そ
れを知っているのは DB ではなく設計者であり、アプリの設計者と別人であっても、
通常はカラムの型やサイズがどうであるのか等の意思疎通をしますよね。ということ
は、充分アプリ側でチェックするロジックを組むことが出来るはずです。

# DB とアプリ相互に信用せず、両方でチェックするという方法もあるでしょう
