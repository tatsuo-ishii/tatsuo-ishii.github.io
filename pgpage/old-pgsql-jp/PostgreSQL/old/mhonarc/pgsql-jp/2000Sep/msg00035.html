> (1)  Ｃ言語でPostgreSQLにアクセスするプログラムを
>  　作成する時には、一般的にPGsetDB()やPGexec()等の
>  　libpq関数のお世話になるのではないんでしょうか？
>  　　それとも、もっと簡便な（高レベルな）手続き
>  　があるんでしようか。

PGsetDB() や PGexec() ってのは、PQsetdb() や PQexec() の間違い
ですかね？ 私は libpq を直接使ってないのでハズしてるかも知れませんが…

libpq は基本ライブラリですが、この手の基本ライブラリは PostgreSQL に
限らず、色々と細かい点を記述する必要があったり、何かと気をつけなければ
ならない面があるので、一般に使い勝手がよくない(生産性が高くない)と
される事が多いのです。

石井さんはそれを言っているに過ぎません。

これは他の DBMS … 例えば Oracle の場合に、「OCI (Oracle Call Interface)
ライブラリを使ってプログラムを書くのは比較的面倒である」というのと
同じです。

libpq や OCI でのプログラミングを全然面倒に感じないという人もいるで
しょうけど、そういう人は別に libpq や OCI を使って書いてればいいんです。

PostgreSQL の場合は、わりと多彩な言語インタフェースがあるので、
単に PostgreSQL を扱いたいというだけであれば、C に拘らなくても
よかろうという事でもあります。

しかし、やはりどうしても C で書きたいけど、libpq や OCI を直接
使うのは面倒だ。という人には、いわゆる埋め込み SQL で書きましょうね。
と、なります。

それが例えば Oracle であれば Pro*C であり、PostgreSQL であれば
ecpg だというわけです。

> (2)  ecpg について具体的な使用方法を紹介したＨＰ
> 　 等がありましたら、教えていただけますか？

HP って Hewlett Packard の事ですかぁ？ とかいうお約束は置いといて、;-p
Linux Japan で笠原さんが書いている、「Linux データベース・レシピ」という
連載で ecpg の解説が載った事があります。

何年何月号かは忘れました。バックナンバーを漁ってみてはいかがでしょう。
笠原さんが、ご自分の Web ページとかで何か書いてたかなぁ…

PostgreSQL のマニュアルにも ecpg に関する記述は当然ありますし、
サンプルもありますので、そのあたりから手をつけるという事に
なるんでしょうね。
